## Findings

<details>
<summary><strong>M-1. Partial-claim truncation can overpay vesting schedule (Fixed)</strong></summary>

> **Review: Valid novel finding. Severity: Medium. The only finding across 5 audits to correctly identify the direction of the integer truncation error.**
>
> Prior audits (Pashov #3, Grimoire S2-A, internal Finding 1a / DC-4) identified the same `claimed / ratePerSec` truncation site but concluded "dust loss to the beneficiary." GPT 5.4 proved the opposite: `claimed % ratePerSec` tokens are paid but the corresponding time is NOT consumed, making that fractional amount reclaimable on the next call.
>
> **Fix applied:** `claimed` is rounded down to whole seconds before transfer, ensuring tokens paid = time consumed exactly:
> ```solidity
> uint64 advance = uint64(claimed / tap.ratePerSec);
> if (advance == 0) revert NothingToClaim();
> claimed = uint256(advance) * uint256(tap.ratePerSec);
> tap.lastClaim += advance;
> ```
> `claimable()` also rounds down to match. The tradeoff: sub-second token dust stays in the DAO until the next whole second accrues — never lost, just deferred.

**Severity:** Medium
**Confidence:** 97
**Location:** `src/peripheral/TapVest.sol:86-95` (formerly L82-92)

### Description

When a claim is capped below `owed`, the contract updates time by:

```solidity
tap.lastClaim += uint64(claimed / tap.ratePerSec);
```

Because this rounds down, the already-paid remainder `claimed % ratePerSec` is not consumed from the vesting clock. As a result, after a partial claim, future claimable amount becomes:

`(owed - claimed) + (claimed % ratePerSec)`

So the contract can preserve extra claimable value and let cumulative claims exceed the intended linear vesting amount.

This is the opposite of the supplied "dust loss" assessment. The code creates **dust credit / overpayment**, not dust loss.

### Attack path

1. A tap is configured with rate `r`.
2. Time elapses so `owed = r * e`.
3. A partial claim occurs with `claimed = c < owed` and `c % r != 0` because DAO balance or allowance is temporarily low.
4. Contract advances only `floor(c / r)` seconds.
5. DAO balance/allowance is replenished later.
6. Another claim can recover the preserved remainder `c % r`.
7. Repeating this across multiple partial claims accumulates excess payout.

### Concrete example

* `ratePerSec = 10`
* `elapsed = 10`
* `owed = 100`
* partial claim with `claimed = 95`

After the claim:

* `lastClaim` advances by `9`
* remaining owed is `10`
* cumulative collectible becomes `95 + 10 = 105`

The extra `5` is created solely by truncation.

### Disproof attempt

* I checked whether line 82 (`min(owed, allowance)`) prevents this. It only caps a single claim; it does not account for already-paid fractional seconds.
* I checked whether allowance consumption at line 95 neutralizes it. It does not; allowance only bounds budget, not vesting correctness.
* I checked against Known Finding 1a / DC-4. They describe the same arithmetic site but with the wrong impact. Since the actual exploit is overpayment rather than dust loss, this is not fully covered by the known tables.

### Fix Applied

`claimed` is now rounded down to the nearest whole-second multiple before transfer:

```diff
  if (claimed == owed) {
      tap.lastClaim = uint64(block.timestamp);
  } else {
-     tap.lastClaim += uint64(claimed / tap.ratePerSec);
+     uint64 advance = uint64(claimed / tap.ratePerSec);
+     if (advance == 0) revert NothingToClaim();
+     claimed = uint256(advance) * uint256(tap.ratePerSec);
+     tap.lastClaim += advance;
  }
```

`claimable()` also rounds down to match:

```diff
  uint256 c = owed < allowance ? owed : allowance;
  if (c > daoBalance) c = daoBalance;
+ if (c < owed) {
+     c = (c / tap.ratePerSec) * tap.ratePerSec;
+ }
  return c;
```

This ensures `tokens paid = time consumed` exactly. Sub-second token dust remains in the DAO until the next whole second of vesting accrues — never lost, just deferred.

### 4) Proportional advance

**Mechanism:** `lastClaim += claimed / ratePerSec` when capped
**Lines:** `src/peripheral/TapVest.sol:82-92`

**Trace**
For a partial claim:

* `owed = ratePerSec * elapsed` at line 72
* `claimed < owed` at lines 82-83
* `lastClaim += claimed / ratePerSec` at line 91

Because integer division truncates, the contract advances by `floor(claimed / ratePerSec)` seconds, not by the exact fraction paid.

That means remaining owed becomes:

`ratePerSec * (elapsed - floor(claimed / ratePerSec))`

Which equals:

`(owed - claimed) + (claimed % ratePerSec)`

So the contract preserves **too much** future claimable amount by `claimed % ratePerSec`.

**Example**

* `ratePerSec = 10`
* `elapsed = 10`
* `owed = 100`
* partial `claimed = 95`

Then:

* line 91 advances by `95 / 10 = 9`
* remaining owed becomes `10 * (10 - 9) = 10`

So total collectible becomes `95 + 10 = 105`, which exceeds the original `100`.

**Conclusion:** **Violated → Fixed**
This defense was broken. The update was not truly proportional; it over-credited vesting after partial claims. Now fixed by rounding `claimed` down to whole seconds.
