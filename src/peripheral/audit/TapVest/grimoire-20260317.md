## Methodology

**Grimoire** is a workflow meta-toolkit for security research. Unlike scanner-style tools, it uses specialized agent roles:

- **Sigil agents** — single-context hunters that receive one attack surface hypothesis and search the codebase for evidence. Each sigil tries to PROVE a vulnerability exists.
- **Familiar agents** — adversarial verifiers that independently read the code and try to DISPROVE each sigil finding. Whatever survives both passes is worth the researcher's time.

## Findings

<details>
<summary><strong>S2-A. Integer division truncation in proportional lastClaim advance (Low — Duplicate)</strong></summary>

**Severity:** Low | **Confidence:** High | **Familiar verdict:** Confirmed

**Location:** `TapVest.sol:82-88`

When `claimed < ratePerSec` (e.g., DAO balance is less than one second's worth of vesting), `claimed / ratePerSec` truncates to 0 and `lastClaim` does not advance. This allows the beneficiary to drain allowance faster than the vesting schedule intends, though total extraction is bounded by the allowance cap.

**Familiar investigation:** Confirmed the truncation is real. The allowance cap (`spendAllowance` decrements `allowance[token][msg.sender]`) provides an absolute ceiling on total extraction. Practical impact is low: for most realistic configurations, the scenario requires the DAO to be nearly empty, at which point remaining value is negligible. Gas costs on mainnet make repeated dust claims economically irrational.

**Prior art:** Duplicate of Pashov #3 (integer division truncation). Acknowledged as dust-level impact in SECURITY.md.

</details>

<details>
<summary><strong>S2-B. Fee-on-transfer tokens cause claim() revert (Informational — Duplicate)</strong></summary>

**Severity:** Informational | **Confidence:** High | **Familiar verdict:** Confirmed

**Location:** `TapVest.sol:90-98`

`spendAllowance` transfers `claimed` tokens to TapVest, but fee-on-transfer tokens deliver `claimed - fee`. The subsequent `safeTransfer(token, beneficiary, claimed)` then reverts due to insufficient balance, permanently DoSing the tap for that token.

**Familiar investigation:** Confirmed. No balance-delta pattern exists in TapVest. The DoS is a revert (not silent fund loss). The DAO chooses the token via `configure()` — this is a configuration concern.

**Prior art:** Duplicate of Winfunc #15 / Pashov #2. Acknowledged in SECURITY.md as DC-3.

</details>

<details>
<summary><strong>S3-A. configure() resets lastClaim forfeiting accrued vesting time (Low — Severity Adjusted)</strong></summary>

**Severity:** Low (adjusted from Medium) | **Confidence:** High | **Familiar verdict:** Severity Adjusted

**Location:** `TapVest.sol:50-55`

When a DAO calls `configure()` on an already-active tap, the entire `TapConfig` is overwritten including `lastClaim = block.timestamp`, silently forfeiting accrued time. Unlike `setRate()` which documents this forfeiture in NatSpec, `configure()` does it implicitly.

**Familiar investigation:** Severity downgraded because: (1) only the DAO itself can call `configure()` via `msg.sender` keying — no external attacker benefits; (2) the beneficiary can self-mitigate by calling `claim()` before the governance proposal executes; (3) granular setters (`setBeneficiary`, `setRate`) exist for parameter changes without full reset; (4) `configure()` is documented as being for `initCalls` during DAO summoning, not reconfiguration.

**Recommendation:** Add NatSpec to `configure()` warning that it resets accrued time (matching `setRate()`'s documentation pattern).

</details>

## Dismissed Findings

| Source | Finding | Reason for Dismissal |
|--------|---------|---------------------|
| Sigil 1 (all) | Reentrancy / CEI violations | `lastClaim` updated before all external calls (L84-88 before L91-98). `spendAllowance` is `nonReentrant` on Moloch. Cross-function reentrancy blocked by `msg.sender` keying. Beneficiary reentry yields `elapsed=0` → `NothingToClaim`. |
| Sigil 3 — claim() griefing | Permissionless claim harms beneficiary | Funds always go to configured beneficiary. Amounts capped by `min(owed, allowance, daoBalance)`. No griefing vector. |
| Sigil 3 — fake DAO | `claim(maliciousDao)` cross-contamination | `taps[dao]` only populated when `dao` calls `configure()`. Fake DAO can only drain itself. |
| Sigil 4 — assembly bugs | Incorrect memory layout / dirty memory | Solady-pattern assembly verified correct: `balanceOf` guards returndatasize, `safeTransfer` handles no-return (USDT), `mstore(0x34,0)` cleanup present. |
| Sigil 4 — stuck ETH | No sweep function for stray ETH | Dismissed by Familiar — duplicate of Known Finding #2. Stray ETH is extractable (not stuck) via fake-DAO pattern. Normal claim flow is atomic. |
