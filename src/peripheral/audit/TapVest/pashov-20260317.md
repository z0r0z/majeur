## Findings

[85] **1. Reentrancy via Malicious DAO in `claim()`**

`TapVest.claim` ¬∑ Confidence: 85 (adjusted from 100)

**Description**
After updating `tap.lastClaim`, `claim()` calls `IMoloch(dao).spendAllowance()` ‚Äî since `dao` is caller-supplied and `configure()` is keyed to `msg.sender`, a malicious contract could configure itself as a "DAO", reenter `claim()` via `spendAllowance`, and return stale allowance values.

**Assessment: Invalid.** The attacker can only populate `taps[attackerContract]` (since `configure` keys to `msg.sender`) and would only drain funds from their own malicious contract ‚Äî there is no victim. On real Moloch DAOs, `spendAllowance` is protected by `nonReentrant`. Duplicate of zellic #8.

---

[85] **2. Fee-on-Transfer Token Accounting Mismatch**

`TapVest.claim` ¬∑ Confidence: 85

**Description**
`claim()` calls `spendAllowance(token, claimed)` which pulls `claimed` tokens to TapVest, then forwards the same `claimed` amount to the beneficiary ‚Äî if `token` is fee-on-transfer, TapVest receives less than `claimed`, causing the subsequent `safeTransfer` to revert (DoS on every claim).

**Assessment: Acknowledged, not fixed.** Duplicate of winfunc #15. TapVest is designed for standard ERC20 tokens; fee-on-transfer tokens are a configuration error. The DoS is self-inflicted by the DAO and recoverable via `configure()` with a different token.

---

[75] **3. Integer Division Truncation Loses Dust on Proportional `lastClaim` Advance**

`TapVest.claim` ¬∑ Confidence: 75

**Description**
When `claimed < owed` (balance-capped), `tap.lastClaim += uint64(claimed / tap.ratePerSec)` truncates the division remainder ‚Äî `claimed % ratePerSec` worth of sub-second time is silently lost on each partial claim, permanently under-compensating the beneficiary by dust amounts.

**Assessment: Acknowledged (dust-level impact).** Maximum loss per partial claim is `ratePerSec - 1` wei of time. For a typical 1 ETH/day rate (~11574 gwei/sec), this is < $0.0001 per occurrence. Adding a carry field would cost an extra storage slot and increase gas for all claims.

[60] **4. Blacklisted Beneficiary Permanently DoSes `claim()`**

`TapVest.claim` ¬∑ Confidence: 60

**Description**
`claim()` pushes funds to `tap.beneficiary` via `safeTransfer`/`safeTransferETH`; if the beneficiary is blacklisted by a token (e.g. USDC), every call to `claim()` reverts until the DAO passes governance to call `setBeneficiary()`.

**Assessment: Acknowledged.** Recoverable by DAO governance. Standard push-payment limitation.
