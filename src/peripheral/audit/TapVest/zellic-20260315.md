## Finding #8 — Reentrant claims before lastClaim update

- **Severity:** Critical
- **Validity:** Invalid

### Description

Zellic reports that `claim` computes `owed` then makes external calls before updating `lastClaim`, allowing reentrancy for multiple payouts.

### Assessment

**Invalid.** The audit incorrectly identifies `IMoloch(dao).allowance` and `balanceOf(token, dao)` as reentrancy vectors to "attacker-controlled contracts." These calls are to:

1. The DAO contract (set by the DAO itself in `configure` via `msg.sender` keying)
2. The token contract (also set by the DAO in `configure`)

Neither is attacker-controlled. Furthermore, `tap.lastClaim = uint64(block.timestamp)` is set at **line 83** *before* external calls at lines 86–93 (`spendAllowance`, `safeTransferETH`/`safeTransfer`). The function follows CEI correctly.

**No fix needed.**

## Finding #2 — Accrued payouts erased via lastClaim reset

- **Severity:** Low
- **Validity:** Valid (Duplicate)

### Description

When `claimed` is capped by allowance or DAO balance, `lastClaim` still advances to `block.timestamp`, permanently discarding any unpaid remainder. `setRate` similarly wipes accrued-but-unpaid amounts when changing the rate.

### Assessment

**Valid, accepted (by design). Duplicate of winfunc #4/13 and webrainsec H-01.**

This is an intentional design trade-off documented across multiple prior audits:

1. **Simplicity over complexity:** Tracking partial accrual debt would add storage, gas, and attack surface. The current design keeps TapVest as a minimal linear stream.
2. **DAO-controlled mitigation:** The DAO controls allowance via governance — if allowance is insufficient, the DAO can top it up before claiming. The `claimable` view function lets anyone check the effective amount before triggering a claim.
3. **`setRate` is DAO-only:** Only the DAO can call `setRate`, so the "erased accrual" on rate change is a conscious governance decision, not an attack vector.
4. **Permissionless claim design:** Any address can trigger `claim`, but funds always go to the configured beneficiary. The "griefing" scenario (calling `claim` during low allowance) only accelerates a partial payout that the beneficiary receives — it doesn't transfer funds to the attacker.

The Zellic-proposed patch (rounding `claimed` to whole seconds and advancing `lastClaim` proportionally) would add complexity and introduce rounding dust but is a reasonable hardening. **Not applied — accepted as by-design.**
