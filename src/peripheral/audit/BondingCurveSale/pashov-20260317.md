~~[85] **1. Hardcoded 1e18 Denominator in `_cost()` Misprices Non-18-Decimal Pay Tokens**~~

`BondingCurveSale._cost` · Confidence: 85

**Status: DISMISSED (false positive).**

The `1e18` divisor normalizes against 18-decimal share/loot amounts, not payToken decimals. Non-18-decimal tokens work correctly out of the box: prices are set in payToken-wei per whole share. For example, 1 USDC/share → `startPrice = 1e6` → `cost = 1e18 * 1e6 / 1e18 = 1e6` (1 USDC). The round-up in `_cost` (`+ denominator - 1`) ensures cost >= 1 payToken-wei for any nonzero purchase, preventing zero-cost dust.

[80] **2. Missing Minimum-Amount Slippage Guard Enables Front-Running**

`BondingCurveSale.buy` · Confidence: 80

**Description**
`buy()` silently caps `amount` to the remaining allowance without a caller-supplied `minAmount` parameter, so a front-runner can deplete available supply and force the victim to receive far fewer tokens at a higher curve price with no on-chain revert.

**Fix**

```diff
- function buy(address dao, uint256 amount) public payable {
+ function buy(address dao, uint256 amount, uint256 minAmount) public payable {
      if (amount == 0) revert ZeroAmount();
      Sale memory s = sales[dao];
      if (s.price == 0) revert NotConfigured();
      if (s.deadline != 0 && block.timestamp > s.deadline) revert Expired();

      uint256 remaining = IMoloch(dao).allowance(s.token, address(this));
      if (amount > remaining) amount = remaining;
      if (amount == 0) revert ZeroAmount();
+     if (amount < minAmount) revert InsufficientPayment();
```

**Response:** Acknowledged — the silent cap is intentional for UX (same pattern as ShareSale). Buyers are protected by `msg.value`: if `amount` is capped down, the cost is lower, and excess ETH is refunded. For ERC20 sales, the buyer pays only for what they receive. The bonding curve means a front-runner pays the lower price and the victim pays the higher price for fewer tokens — but the victim's tx reverts if `msg.value < cost` (ETH) or they haven't approved enough (ERC20). Front-running risk is inherent to any public sale. **Status: Accepted (by design)**

[75] **3. Fee-on-Transfer Payment Token Underpays the DAO**

`BondingCurveSale.buy` · Confidence: 75

**Description**
When a fee-on-transfer ERC20 is configured as `payToken`, `safeTransferFrom` transfers `cost` from buyer to DAO but the DAO only receives `cost * (1 - fee)`, while the buyer receives the full `amount` of shares/loot — the DAO is systematically underpaid for every sale.
