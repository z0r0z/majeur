## Findings

**LOW-01. `initShares.length` not validated against `initHolders.length`**

Confidence: High

**Description**
The wrapper does not locally enforce `initShares.length == initHolders.length`. The only array-length check is for `initLoot`. An `initShares` / `initHolders` mismatch would pass `_validate` and `_buildCalls`, only to revert downstream.

> **Response — Accepted (no fix needed).** `Moloch.init()` at L221 enforces `require(initHolders.length == initShares.length, LengthMismatch())`. The invariant is guaranteed by the downstream contract. SafeSummoner is a deployment convenience wrapper — duplicating every downstream check is not a design goal. The `initLoot` check exists because loot minting is built *inside* SafeSummoner (not forwarded to Summoner), so the wrapper must validate it locally. Previously identified as a design observation in Archethect MH-008 (DISCARD).

---

**LOW-02. `saleBurnDeadline` always targets shares, even when `saleIsLoot = true`**

Confidence: High

**Description**
When `saleBurnDeadline > 0`, the burn permit always targets `_predictShares(dao)` regardless of `saleIsLoot`. If a DAO is configured with `saleIsLoot = true` and `saleBurnDeadline > 0`, the burn permit targets shares instead of loot — the unsold loot would not be burnable via the generated permit.

> **Response — Confirmed, fix applied.** This is a genuine wrapper correctness bug. The burn permit should target the sale token. See Hardening Patches below.

---

**LOW-03. Seed module allows duplicate sentinel mints for the same token**

Confidence: Medium

**Description**
No check prevents both seed sides from resolving to the same token (e.g. `tokenA = address(1)` and `tokenB = address(1)`), which would mint the same asset twice and create a same-token LP pair.

> **Response — Accepted (no fix needed).** The downstream `LPSeedSwapHook.configure()` validates that `tokenA != tokenB` after sentinel resolution. If both resolve to the same address, deployment reverts during the init call. SafeSummoner intentionally delegates module-level parameter validation to the singletons themselves (documented as Design Choice DC-2). Adding redundant checks in the wrapper would couple it to each singleton's validation logic.

---

**INFO-01. `renderer == address(0)` documented as "default" but not enforced locally**

Confidence: High

**Description**
The NatSpec states `renderer = address(0)` uses the default renderer, but the wrapper passes it through directly. Only preset helpers hardcode `RENDERER`.

> **Response — Accepted (no fix needed).** `Moloch.init()` at L228 handles this: `if (_renderer != address(0)) renderer = _renderer`. When zero is passed, the DAO keeps its default renderer (set during implementation deployment). The NatSpec correctly describes the end-to-end behavior. The wrapper deliberately exposes the full parameter to non-preset callers for flexibility. This is the same delegation pattern as LOW-01.

## Positive Security Properties Noted

The reviewer confirmed the following validated invariants:

- rejects zero initial holder count
- rejects zero `proposalThreshold`
- rejects zero `proposalTTL`
- rejects `quorumBps > 10_000`
- rejects `timelockDelay >= proposalTTL`
- rejects futarchy with zero quorum
- rejects futarchy with zero cap
- rejects minting sale + dynamic-only quorum
- rejects sale configuration with zero price
- rejects rollback guardian without rollback singleton
- rejects seed sale gating when no sale module is configured
- rejects `SafeConfig.saleActive` and `SaleModule` being used simultaneously
- caps `_defaultThreshold(...)` to `uint96.max`
