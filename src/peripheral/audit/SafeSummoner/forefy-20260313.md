## Round 1: Security Expert 1 — Systematic Code Review

### Scope Analysis

- **Core factory:** SafeSummoner (validation, call building, CREATE2, multicall, DAO deployment via Summoner singleton)
- **No token contracts** — SafeSummoner does not hold, mint, or transfer tokens itself
- **No storage** — zero state variables, zero persistent storage
- **7 external functions:** `safeSummon`, `safeSummonDAICO`, `summonStandard`, `summonFast`, `summonFounder`, `create2Deploy`, `multicall`
- **5 internal builders:** `_buildCalls`, `_buildModuleCalls`, `_buildLootMints`, `_validate`, `_validateModules`

### Highest-Risk Function Analysis

**1. Reentrancy (fv-sol-1)**

- `multicall()` L208: `address(this).delegatecall(data[i])` — delegatecall in a loop
- SafeSummoner has **zero storage variables** — no persistent state to corrupt via reentrancy
- No `nonReentrant` guard exists, but none is needed — stateless contract
- `create2Deploy()` L235: `create2()` — deployed constructor could call back to SafeSummoner. Same argument: no storage to corrupt.
- Cross-referenced fv-sol-1-c9: transient storage mutex — **not applicable**, no transient storage used, no storage to protect

> **Review:** Reentrancy is structurally impossible. SafeSummoner has zero storage variables — no state to corrupt regardless of call path. The stateless architecture eliminates fv-sol-1 by construction.

**2. Precision Errors (fv-sol-2)**

- `_defaultThreshold()` L1060: `uint256 t = total / 100` — integer division with floor
- For small initial supplies (e.g., total = 99), `t = 0`, caught by `if (t == 0) t = 1` floor guard at L1061
- For extreme supplies (>7.9e30), `uint96(t)` downcast — caught by saturating cap at L1062: `if (t > type(uint96).max) t = type(uint96).max`
- No mulDiv or ERC4626-style share accounting anywhere in SafeSummoner

> **Review:** Precision handling is correct. Floor guard prevents zero threshold. Saturating cap prevents uint96 overflow. No fv-sol-2 vulnerability.

**3. Arithmetic Errors (fv-sol-3)**

- All arithmetic uses Solidity 0.8.30 default checked mode — no `unchecked` blocks
- `_defaultThreshold()` L1058: `total += initShares[i]` — checked addition, reverts on overflow
- `_defaultThreshold()` L1062: saturating cap handles the uint96 downcast safely
- Assembly blocks at L211-213 (`multicall` revert bubbling) and L232-235 (`create2Deploy` creation) — no arithmetic operations, just `REVERT`, `MLOAD`, `CALLDATACOPY`, `CREATE2`
- Cross-referenced fv-sol-3-c5: assembly arithmetic — **no arithmetic in assembly blocks**

> **Review:** All arithmetic is checked. Assembly blocks contain no arithmetic operations. No fv-sol-3 vulnerability.

**4. Access Control (fv-sol-4)**

- All 7 external functions are permissionless (`public` with no modifiers)
- SafeSummoner is a **stateless factory** — no state to protect via access control
- `msg.sender` is only used in `summonFounder()` L367 to set `h[0] = msg.sender` as initial holder — correct and intentional
- `create2Deploy()` salt is not bound to `msg.sender` — CREATE2 address is sender-independent
- Cross-referenced fv-sol-4-c9: CREATE2 squatting — attacker frontruns `create2Deploy` with same (bytecode, salt), deploys the victim's contract at the predicted address with correct bytecode. No code injection possible (CREATE2 address includes bytecode hash). Victim retries without the create2 step. (Duplicate: Pashov #2)

> **Review:** Permissionless access is correct for a stateless factory. CREATE2 squatting is a known benign frontrun — attacker pays gas to deploy the victim's contract. No fv-sol-4 vulnerability.

**5. Logic Errors (fv-sol-5)**

**5a. msg.value Reuse in Multicall (fv-sol-5-c7)**

- `multicall()` L206: `public payable` — accepts ETH
- L209: `address(this).delegatecall(data[i])` — each sub-call sees the same `msg.value`
- **Unlike Moloch.sol's `multicall` which is NOT payable**, SafeSummoner's `multicall` IS payable
- Per fv-sol-5-c7: msg.value is reused across delegatecall iterations
- NatSpec at L203-205 explicitly documents this: "msg.value is shared across all calls — callers sending ETH must ensure only one sub-call consumes it"
- Self-contained risk — only the caller's own ETH is at stake, caller controls the data array
- No external attacker can exploit this

> **Review:** Documented design tradeoff. Standard delegatecall-multicall pattern (Uniswap V3 Router, Seaport). Caller controls both data array and ETH sent. (Duplicate: SCV Scan #1, Archethect DT-02)

**5b. Force ETH Injection (fv-sol-5-c8)**

- SafeSummoner has no ETH-denominated accounting and no `address(this).balance` reads
- ETH received via `msg.value` is forwarded atomically to `SUMMONER.summon{value: msg.value}()`
- Per fv-sol-5-c8 false positive: contract does not use `address(this).balance` for accounting

> **Review:** Not vulnerable. SafeSummoner forwards ETH atomically and does not rely on its own balance. No fv-sol-5-c8 issue.

**5c. Conditional Logic in Validation (fv-sol-5-c2)**

- `_validate()` L634-663: 8 validation checks with custom errors
- `_validateModules()` L666-683: 3 module-specific checks
- Verified all condition branches:
  - `quorumBps > 10_000` → reverts correctly (not `>=`, since 10000 = 100% is valid)
  - `timelockDelay > 0 && proposalTTL <= timelockDelay` → correctly prevents TTL = timelock (edge case where proposal expires exactly at queue end)
  - `saleActive && saleMinting && quorumBps > 0 && quorumAbsolute == 0` → correctly identifies KF#2 condition
  - `seed.gateBySale && sale.singleton == address(0)` → correctly prevents orphaned gate

> **Review:** All conditional logic is correct. No off-by-one or inverted condition issues. No fv-sol-5-c2 vulnerability.

**6. Unchecked Returns (fv-sol-6)**

- `SUMMONER.summon{value: msg.value}(...)` — returns `address`. Called at L299, L401, L459, L551, L580. Return value is captured in `dao` and returned to caller.
- `address(this).delegatecall(data[i])` L209 — return value `ok` is checked at L210, reverts if false with error bubbling
- `create2()` L235 — return value checked at L237: `if (deployed == address(0)) revert Create2Failed()`
- No low-level `call` without return checking

> **Review:** All external calls have their return values checked. No fv-sol-6 vulnerability.

**6b. Return Data Bomb (fv-sol-6-c8)**

- `multicall` delegatecall L209: `(bool ok, bytes memory result)` — result is stored in memory array
- Malicious sub-call could return large data. However, sub-calls target `address(this)` — SafeSummoner's own functions. The caller controls the data array and chooses which functions to call.
- Per fv-sol-6-c8 false positive: "Call target is self (hardcoded address(this))"

> **Review:** Not vulnerable. Delegatecall targets self only. Caller controls all sub-calls.

**7. Unbounded Loops (fv-sol-9)**

- `_buildCalls()` L835: iterates `extra.length` — caller-supplied
- `_buildLootMints()` L1022, L1027: iterates `loot.length` — caller-supplied
- `_buildModuleCalls()`: fixed-count (max 7 calls for all three modules)
- `_defaultThreshold()` L1057: iterates `initShares.length` — caller-supplied
- `multicall()` L208: iterates `data.length` — caller-supplied
- `safeSummonDAICO()` L447-454: three merge loops — caller-supplied arrays
- All loops iterate caller-supplied data. Only the caller's own transaction gas is consumed.
- Per fv-sol-9 false positive: "Bounded by caller-supplied arrays, gas cost borne by caller"

> **Review:** All loops are caller-bounded. No unbounded storage iteration. No fv-sol-9 vulnerability.

## Round 2: Security Expert 2 — Economic & Integration Focus

### Independent Protocol Analysis

**1. Module Wiring Integrity — Allowance/Configure Ordering**

The `_buildModuleCalls` function generates initCalls executed by the newly deployed DAO. The order is:

1. SaleModule: `setAllowance(sale.singleton, saleToken, cap)` → `sale.configure(...)`
2. TapModule: `setAllowance(tap.singleton, token, budget)` → `tap.configure(...)`
3. SeedModule: `mint(dao, amount)` → `setAllowance(singleton, tokenA, amountA)` → `setAllowance(singleton, tokenB, amountB)` → `seed.configure(...)`

**Analysis:** Allowance is always set before `configure` for each module. SeedModule correctly mints sentinel tokens before setting allowances (tokens must exist before they can be allowed). The order is: mint → allow → configure. This is the correct dependency chain.

> **Review:** Module wiring order is correct. No ordering vulnerability.

**2. Sentinel Address Collision Analysis**

SafeSummoner uses multiple sentinel address conventions:

| Sentinel | Context | Meaning |
|---|---|---|
| `address(dao)` | SaleModule allowance | Mint shares (Moloch allowance sentinel) |
| `address(1007)` | SaleModule allowance | Mint loot (Moloch allowance sentinel) |
| `address(1)` | SeedModule token | DAO shares (resolved to predicted ERC20) |
| `address(2)` | SeedModule token | DAO loot (resolved to predicted ERC20) |
| `address(0)` | Multiple fields | Skip/default/ETH depending on context |

**Analysis:** No sentinel collision. The SaleModule and SeedModule sentinel spaces are disjoint: SaleModule sentinels (`address(dao)`, `address(1007)`) are never exposed to SeedModule, and vice versa. The `_resolveSaleToken` and `_resolveSeedToken` functions handle their own sentinel spaces independently.

Edge case: if `dao == address(1)` or `dao == address(2)`, `_resolveSeedToken` would misinterpret a real DAO address as a sentinel. However, DAO addresses are CREATE2-derived from `keccak256(...)` — the probability of colliding with `address(1)` or `address(2)` is ~2^-160. Not exploitable.

> **Review:** Sentinel namespaces are cleanly separated. No collision risk.

**3. Validation Completeness — Cross-Module Constraints**

Analyzed which configurations are validated vs. which are passed through unchecked:

| Parameter | Validated? | How |
|---|---|---|
| proposalThreshold | ✓ | Must be > 0 |
| proposalTTL | ✓ | Must be > 0 |
| timelockDelay | ✓ | Must be < TTL |
| quorumBps | ✓ | Must be ≤ 10000 |
| futarchy + quorum | ✓ | KF#17 |
| futarchy + cap | ✓ | KF#3 |
| minting sale + dynamic quorum | ✓ | KF#2 (both SafeConfig and SaleModule paths) |
| sale price | ✓ | Must be > 0 when active |
| seed gate + sale | ✓ | Gate requires sale |
| SafeConfig.sale + SaleModule | ✓ | ModuleSaleConflict |
| TapModule budget/rate | ✗ | Passed through unchecked |
| SeedModule amounts | ✗ | Passed through unchecked |
| rollbackExpiry | ✗ | Passed through unchecked |
| extraCalls | ✗ | Passed through unchecked (by design) |

**Analysis:** TapModule and SeedModule parameters are not validated by SafeSummoner — they rely on the downstream singletons' own validation. This is correct for a wrapper pattern: SafeSummoner validates governance invariants (KF findings), while peripheral modules validate their own domain constraints.

> **Review:** Validation covers all governance invariants. Module-specific params correctly delegated to singletons.

**4. RollbackGuardian Permit Scope**

SafeSummoner wires two one-shot permits for the RollbackGuardian singleton:
1. `bumpConfig()` — invalidates all in-flight proposals (emergency reset)
2. `setAutoFutarchy(0, 0)` — disables futarchy (emergency kill)

Both permits use `count = 1` (one-shot). After spending, the guardian cannot repeat the action without a new governance proposal granting a new permit.

**Analysis:** The permit scope is tight: only `bumpConfig` and `killFutarchy`, each usable once. No escalation path — the singleton is the spender, not the guardian EOA directly. The guardian triggers the singleton, which spends the permit.

> **Review:** RollbackGuardian permits are correctly scoped. One-shot, non-escalating.

**5. summonFounder Preset — Hardcoded Values**

`summonFounder()` L360-404 creates a single-owner DAO with:
- 10M shares to `msg.sender`
- 1% proposal threshold (100,000e18)
- 1-day TTL
- No timelock
- 100 bps quorum (1%)
- Ragequittable

**Analysis:** The preset inlines `_predictDAO` logic because memory arrays can't be passed to calldata. The inlined prediction matches `_predictDAO` exactly (same salt derivation, same creation code). The quorum (1%) and threshold (1%) are coherent — the single founder holds 100% of shares and can always propose and pass votes unilaterally.

Edge case: `summonFounder` does not call `_validate` — it bypasses validation entirely. However, all hardcoded values satisfy the validation constraints:
- `proposalThreshold = 100_000e18 > 0` ✓
- `proposalTTL = 1 days > 0` ✓
- `timelockDelay = 0` — no TTL comparison needed ✓
- `quorumBps = 100 ≤ 10_000` ✓
- No futarchy, no sale — remaining checks N/A ✓

> **Review:** summonFounder hardcoded values are safe. Validation bypass is acceptable because all invariants are satisfied by construction.

### Oversight Analysis of Expert 1

Expert 1's analysis is thorough. I concur with all findings and false positive determinations. Additional notes:

- **_validate takes SafeConfig as memory, not calldata** (L634): When called from `_summonPreset` and `_summonDAICOPreset`, the config is a memory struct. When called from `safeSummon` and `safeSummonDAICO`, the calldata config is implicitly copied to memory for the function call. This is gas-correct — the copy is unavoidable for internal functions that need to work with both memory and calldata structs.

- **extraCalls passthrough trust model**: `extraCalls` are caller-supplied and appended to initCalls without inspection. These execute in the new DAO's context during `init()`. The caller is the DAO deployer — they're configuring their own DAO. No trust boundary violation.

### Finding-by-Finding Triager Validation

**Spot 1: msg.value Shared Across Multicall Delegatecalls (fv-sol-5-c7)**

- **Technical Disproof Attempt:** Each `delegatecall` in the loop sees `msg.value`. If caller batches two `safeSummon{value: 1 ether}` calls, the first forwards 1 ETH, the second attempts the same. However, the ETH is forwarded via `SUMMONER.summon{value: msg.value}(...)` — the actual ETH transfer reduces `address(this).balance`. The second call would fail with insufficient balance, NOT silently succeed.
- **Economic Disproof Attempt:** Only caller's own ETH at risk. Caller controls the data array. No external attacker can inject sub-calls.
- **Verdict:** DISMISSED — self-contained, documented, caller-controlled. Standard industry pattern.
- **Bounty Assessment:** $0 — Documented design tradeoff, not a vulnerability.

**Spot 2: create2Deploy Salt Not Bound to msg.sender (fv-sol-4-c9)**

- **Technical Disproof Attempt:** Attacker frontruns `create2Deploy(bytecode, salt)` → deploys the same contract at the predicted address with correct bytecode (CREATE2 address includes bytecode hash). Victim's call reverts, but the contract exists.
- **Economic Disproof Attempt:** Attacker pays gas. Victim gets their contract deployed for free. No code injection, no fund theft. Victim retries multicall without the create2 step or uses a different salt.
- **Verdict:** DISMISSED — benign frontrun, attacker subsidizes victim.
- **Bounty Assessment:** $0 — no exploitation scenario exists.

**Spot 3: _defaultThreshold uint96 Saturating Cap (fv-sol-3)**

- **Technical Disproof Attempt:** Saturating cap at L1062 prevents overflow. The scenario requires total shares > 7.9e30, which breaks Moloch.sol's entire uint96-based governance system upstream.
- **Economic Disproof Attempt:** No sane deployment has >7.9e28 total shares. If it did, vote tallies, quorum checks, and delegation accounting all wrap — the DAO is fundamentally broken before SafeSummoner's threshold matters.
- **Verdict:** DISMISSED — patched defensively, precondition is catastrophic. No practical exploitation.
- **Bounty Assessment:** $0 — patched, non-issue in practice.

**Spot 4: summonFounder Bypasses _validate**

- **Technical Disproof Attempt:** All hardcoded values satisfy every validation constraint. Enumerated each check — all pass. No configuration path allows a caller to inject invalid values.
- **Verdict:** DISMISSED — invariants satisfied by construction.
- **Bounty Assessment:** $0 — not a vulnerability.

**Spot 5: Unbounded Caller-Supplied Arrays (fv-sol-9)**

- **Technical Disproof Attempt:** All loops iterate caller-supplied arrays. Gas cost borne by caller only.
- **Verdict:** DISMISSED — caller self-DoS only.
- **Bounty Assessment:** $0

**Spot 6: extraCalls Execute in DAO Context**

- **Technical Disproof Attempt:** `extraCalls` are appended to initCalls and executed by the new DAO during `init()`. Caller is the deployer — they're configuring their own DAO. No trust boundary violation.
- **Verdict:** DISMISSED — by design, caller configures their own DAO.
- **Bounty Assessment:** $0

### Key Observations

1. **Stateless architecture is the primary defense.** SafeSummoner has zero storage variables, eliminating reentrancy, storage corruption, access control, upgrade collision, and state desynchronization vulnerability classes by construction. 3. **Validation logic is complete and correct.** All known Moloch governance attack vectors (KF#2, KF#3, KF#11, KF#17) are enforced with appropriate reverts. Module-specific constraints are correctly validated. Cross-module conflicts (SafeConfig.sale vs SaleModule) are caught.

4. **Module wiring is correctly ordered.** Mint → allowance → configure dependency chain is respected for all three module types. Sentinel address namespaces are cleanly separated.

### Novel Findings

**Zero.** All 3 informational observations have been independently identified by 2-4 prior auditors. The Forefy three-layer reading order (governance protocol context → fv-sol deep theory → Solidity tricks) confirmed the existing audit coverage without surfacing new issues.

### What the Framework Got Right

The fv-sol knowledge base correctly identified SafeSummoner's stateless architecture as eliminating most vulnerability classes at the structural level. The false positive criteria for each category (fv-sol-1: "no storage to corrupt", fv-sol-5-c7: "documented behavior", fv-sol-5-c8: "no balance-based accounting", fv-sol-9: "caller-bounded arrays") provided precise, auditable dismissal justifications. The conservative severity calibration (round DOWN) and triager budget-protection role correctly prevented severity inflation across all 3 informational items.

## Responses

### Finding 1 — msg.value Shared Across Multicall Delegatecalls

**Response — Accepted (no fix needed).** Standard delegatecall-multicall behavior, identical to Uniswap V3 Router, Seaport, and other production contracts. NatSpec at L203-205 explicitly documents the behavior. The caller controls both the data array and the ETH sent — no external attacker can exploit this. Only the caller's own misconfiguration could cause issues, and they'd see a revert (insufficient balance for second sub-call).

### Finding 2 — create2Deploy Salt Not Bound to msg.sender

**Response — Accepted (no fix needed).** The frontrunning scenario is benign: attacker deploys the victim's contract (correct bytecode) at the predicted address. Victim retries without the create2 step. Binding salt to msg.sender would break sender-independent address prediction, complicating cross-EOA and contract-based deployment flows.

### Finding 3 — _defaultThreshold uint96 Saturating Cap

**Response — Patched.** Saturating cap added defensively at L1062: `if (t > type(uint96).max) t = type(uint96).max`. Non-issue in practice — Moloch.sol uses uint96 throughout (vote tallies, quorum, threshold), so total shares exceeding uint96.max breaks the entire governance system upstream.
