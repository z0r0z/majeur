## Methodology

**Grimoire** is a workflow meta-toolkit for security research. Unlike scanner-style tools, it uses specialized agent roles:

- **Sigil agents** — single-context hunters that receive one attack surface hypothesis and search the codebase for evidence. Each sigil tries to PROVE a vulnerability exists.
- **Familiar agents** — adversarial verifiers that independently read the code and try to DISPROVE each sigil finding. Whatever survives both passes is worth the researcher's time.

The full workflow was executed manually using Grimoire's agent prompts (`agents/sigil.md`, `agents/familiar.md`) and the Summon skill's crown jewels reference (`skills/summon/references/domain-crown-jewels.md`).

## Sigil 1: Reentrancy + Storage Corruption

### Hypothesis

SafeSummoner uses `delegatecall` (multicall) and `create2` (constructor callbacks). Can an attacker corrupt persistent state via reentrant calls?

### Investigation

1. **Storage inventory:** Zero state variables declared. No `mapping`, `uint`, `address`, `bool`, or `struct` at contract level. `constructor() payable {}` is empty.
2. **Assembly scan:** Two assembly blocks — `multicall` L211 (REVERT only) and `create2Deploy` L232 (MLOAD, CALLDATACOPY, CREATE2). Neither performs `SSTORE` or `TSTORE`.
3. **Delegatecall target:** `address(this).delegatecall(data[i])` — hardcoded self-reference. Runs SafeSummoner's own functions in SafeSummoner's storage context. But there is no storage to write to.
4. **CREATE2 constructor callback:** Deployed contract's constructor runs during `create2()`. Could call back to SafeSummoner. Same conclusion: no storage to corrupt.

### Result

**0 findings.** The reentrancy attack surface does not exist. SafeSummoner has zero persistent storage — there is nothing for a reentrant call to corrupt, read-before-write, or desynchronize.

## Sigil 2: Validation Logic + Governance Invariants

### Hypothesis

SafeSummoner's `_validate` and `_validateModules` enforce downstream DAO safety invariants. Can a caller bypass these checks to deploy a misconfigured DAO?

### Investigation

**2a. Validation completeness scan:**

| Invariant | Check Location | Bypass Path? |
|-----------|---------------|-------------|
| proposalThreshold > 0 (KF#11) | `_validate` L636 | No — reverts `ProposalThresholdRequired` |
| proposalTTL > 0 | `_validate` L637 | No — reverts `ProposalTTLRequired` |
| quorumBps ≤ 10000 | `_validate` L638 | No — reverts `QuorumBpsOutOfRange` |
| TTL > timelockDelay | `_validate` L641 | No — reverts `TimelockExceedsTTL` |
| futarchy + quorum (KF#17) | `_validate` L646 | No — reverts `QuorumRequiredForFutarchy` |
| futarchy + cap (KF#3) | `_validate` L653 | No — reverts `FutarchyCapRequired` |
| minting sale + dynamic quorum (KF#2) | `_validate` L658 + `_validateModules` L675 | No — both paths covered |
| sale price > 0 | `_validate` L662 + `_validateModules` L673 | No — both paths covered |
| seed gate requires sale | `_validateModules` L680 | No — reverts `SeedGateWithoutSale` |
| SafeConfig.sale + SaleModule conflict | `safeSummonDAICO` L431 | No — reverts `ModuleSaleConflict` |

**2b. summonFounder validation bypass:**

`summonFounder()` does not call `_validate`. It uses hardcoded values. Enumerated each constraint:
- `proposalThreshold = 100_000e18 > 0` ✓
- `proposalTTL = 1 days > 0` ✓
- `timelockDelay = 0` — no TTL comparison needed ✓
- `quorumBps = 100 ≤ 10_000` ✓
- No futarchy, no sale — remaining checks N/A ✓

All invariants satisfied by construction. No bypass.

**2c. extraCalls can override validated settings:**

`extraCalls` are appended after `_buildCalls` output and execute in the new DAO's `init()` context. A caller could include a `setProposalThreshold(0)` call in `extraCalls`, overriding the validated threshold.

| S2-1 | summonFounder bypasses _validate | Medium |

## Sigil 3: Economic + Module Wiring + Sentinels

### Hypothesis

SafeSummoner wires allowances, mints, and configures three peripheral modules (ShareSale, TapVest, LPSeedSwapHook). Can misordering, sentinel confusion, or allowance miscalculation lead to fund loss?

### Investigation

**3a. Module call ordering:**

`_buildModuleCalls` generates initCalls in this order:
1. SaleModule: `setAllowance(singleton, token, cap)` → `configure(token, payToken, price, deadline)`
2. TapModule: `setAllowance(singleton, token, budget)` → `configure(token, beneficiary, rate)`
3. SeedModule: `mint(dao, amount)` [if sentinel] → `setAllowance(singleton, tokenA, amountA)` → `setAllowance(singleton, tokenB, amountB)` → `configure(...)`

Dependency chain: mint → allow → configure. Correct for all three modules.

**3b. Sentinel namespace collision:**

| Address | SaleModule meaning | SeedModule meaning |
|---------|-------------------|-------------------|
| `address(dao)` | Mint shares sentinel | N/A (never used as seed token) |
| `address(1007)` | Mint loot sentinel | N/A |
| `address(1)` | N/A | Shares (resolved to ERC20) |
| `address(2)` | N/A | Loot (resolved to ERC20) |
| `address(0)` | ETH / skip | ETH / skip |

Namespaces are disjoint. `_resolveSaleToken` and `_resolveSeedToken` handle their own spaces independently. Edge case: `dao == address(1)` — probability ~2^-160, not exploitable.

**3c. Allowance amount consistency:**

SeedModule sets allowance equal to the seed amount for both tokens. If `amountA = 1000e18` and `amountB = 500e18`, the hook gets exactly those allowances. No over-allowance, no under-allowance.

SaleModule sets allowance to `sale.cap`. When `sale.minting = true`, the allowance sentinel (`address(dao)` or `address(1007)`) is used — Moloch interprets this as mint authorization. Correct.

### Raw Findings

| # | Finding | Sigil Severity |
|---|---------|---------------|
| S3-1 | Module wiring ordering could be documented more explicitly | Informational |
| S3-2 | SeedModule sentinel collision with DAO address at address(1)/address(2) | Low |
| S3-3 | TapModule/SeedModule parameters not validated by SafeSummoner | Informational |

## Sigil 4: Access Control + CREATE2 + Multicall

### Hypothesis

SafeSummoner is fully permissionless with CREATE2 deployment and delegatecall multicall. Can an attacker exploit the lack of access control, frontrun deployments, or abuse multicall?

### Investigation

**4a. Permissionless factory:**

All 7 public functions have no access restriction. SafeSummoner has no state — nothing to protect. Each call creates a new DAO where the caller's addresses are baked into the CREATE2 salt. No impersonation possible.

**4b. CREATE2 frontrunning:**

`create2Deploy(bytecode, salt)` — salt is not bound to `msg.sender`. Attacker can frontrun with same params. Result: victim's contract deployed at predicted address with correct bytecode (CREATE2 address includes bytecode hash). Attacker pays gas. Victim retries without the create2 step.

**4c. multicall msg.value sharing:**

`multicall` is `payable`. `delegatecall` preserves `msg.value` across iterations. Each sub-call sees the same `msg.value`. However, actual ETH transfer (via `SUMMONER.summon{value: msg.value}`) reduces `address(this).balance`. Second ETH-consuming sub-call would revert on insufficient balance. Self-contained: caller controls both data array and ETH.

**4d. multicall delegatecall target:**

`address(this).delegatecall(data[i])` — target is hardcoded `address(this)`. Cannot be redirected. Data controls which function is called, but all functions are SafeSummoner's own permissionless functions. No `onlyDAO` or privileged functions exist to exploit.

**4e. Predicted address reliance:**

All summon functions predict the DAO address before deployment (`_predictDAO`). If the prediction is wrong, initCalls would target the wrong address and fail. The prediction uses the same creation code and salt as the Summoner — correctness is guaranteed by construction.

### Raw Findings

| # | Finding | Sigil Severity |
|---|---------|---------------|
| S4-1 | CREATE2 salt not bound to msg.sender — frontrun DoS | Medium |
| S4-2 | multicall msg.value shared across delegatecalls | Medium |
| S4-3 | Permissionless factory — no access control | Low |
| S4-4 | multicall delegatecall could invoke any SafeSummoner function | Low |
| S4-5 | Address prediction correctness unverified at runtime | Informational |

## Familiar Triage (3 agents, parallel)

### Familiar 1 — Sigil 2 + Sigil 3 Triage

**S2-1: summonFounder bypasses _validate**

- **Disproof attempt:** Enumerated all 10 validation constraints against hardcoded values. All satisfied by construction. No caller-controlled parameter can produce a value that violates any constraint. The function takes only `orgName`, `orgSymbol`, `orgURI`, `salt` — none of which affect governance parameters.
- **Verdict:** DISMISSED — invariants satisfied by construction, no bypass possible.

**S2-2: extraCalls can override validated config**

- **Disproof attempt:** `extraCalls` are caller-supplied and labeled "advanced use" in NatSpec (L269). The caller is deploying their own DAO — there is no trust boundary violation. Direct summoning via `SUMMONER.summon()` bypasses SafeSummoner entirely, achieving the same result. SafeSummoner is advisory, not a security boundary.
- **Verdict:** CONFIRMED as Design Choice — documented foot-gun, not a vulnerability. (Duplicate: Grimoire Moloch audit L-04)

**S3-1: Module wiring ordering documentation**

- **Disproof attempt:** The ordering is correct and the code is self-documenting (comments at L877: "Order: SaleModule → TapModule → SeedModule (mints → allowances → configure)"). No vulnerability.
- **Verdict:** DISMISSED — correct and documented.

**S3-2: SeedModule sentinel collision with DAO address**

- **Disproof attempt:** DAO addresses are derived from `keccak256(abi.encodePacked(0xff, SUMMONER, salt, codeHash))`. The probability of a DAO address being `address(1)` or `address(2)` is 2^-160. Not exploitable.
- **Verdict:** DISMISSED — probability is negligible.

**S3-3: TapModule/SeedModule params not validated**

- **Disproof attempt:** SafeSummoner validates governance invariants. Module-specific constraints (tap rate, seed amounts, deadlines) are domain-specific to each singleton and validated by the singletons' own `configure` functions. Duplicating validation would create maintenance burden and coupling.
- **Verdict:** CONFIRMED as Design Choice — correct delegation of responsibility.

### Familiar 2 — Sigil 4 Triage

**S4-1: CREATE2 salt not bound to msg.sender**

- **Disproof attempt:** Attacker frontruns `create2Deploy(bytecode, salt)`. Result: attacker deploys the victim's contract with correct bytecode at the predicted address (CREATE2 address = f(bytecode_hash, salt, deployer)). Victim's tx reverts but the contract exists. Victim retries multicall without the create2 step, or uses a different salt. No code injection (address includes bytecode hash). Attacker pays gas to deploy victim's contract.
- **Verdict:** CONFIRMED as Informational — benign frontrun, attacker subsidizes victim. (Duplicate: Pashov AI #1 Finding 2, Archethect DT-01, Forefy #2)

**S4-2: multicall msg.value shared across delegatecalls**

- **Disproof attempt:** Standard delegatecall-multicall pattern (Uniswap V3 Router, Seaport). NatSpec at L203-205 explicitly documents the behavior. Caller controls both the data array and the ETH sent. Second ETH-consuming sub-call would revert on insufficient balance — no silent double-spend. Only caller's own ETH at risk.
- **Verdict:** CONFIRMED as Informational — documented design tradeoff, industry standard. (Duplicate: SCV Scan #2 Finding 1, Archethect DT-02, Forefy #1)

**S4-3: Permissionless factory**

- **Disproof attempt:** SafeSummoner has zero storage. No state to protect. All functions create new state on external contracts (deployed DAOs). Share ownership baked into CREATE2 salt. No impersonation possible.
- **Verdict:** DISMISSED — stateless factory, access control would add complexity without security benefit.

**S4-4: multicall delegatecall could invoke any SafeSummoner function**

- **Disproof attempt:** Target is hardcoded `address(this)`. All SafeSummoner functions are already permissionless. Invoking `safeSummon` via multicall delegatecall is identical to calling it directly — no privilege escalation. No `onlyDAO` or admin functions exist.
- **Verdict:** DISMISSED — no privileged functions to abuse.

**S4-5: Address prediction correctness**

- **Disproof attempt:** `_predictDAO` uses the same salt derivation (`keccak256(abi.encode(holders, shares, salt))`) and the same EIP-1167 creation code as the Summoner. The prediction is correct by construction — it replicates the CREATE2 address formula exactly. If it were wrong, `_buildCalls` would target a nonexistent address and the calls would fail during `init()`.
- **Verdict:** DISMISSED — correct by construction, failures caught at runtime.

## Finding Responses

### [I-01] CREATE2 Salt Not Bound to msg.sender — Duplicate

**Severity:** Informational | **Confidence:** High | **Familiar verdict:** Confirmed (benign frontrun)
**Location:** `SafeSummoner.sol` L227-238 — `create2Deploy()`

The sigil identified that `create2Deploy` uses caller-supplied salt without binding to `msg.sender`, enabling frontrun deployment at the predicted address. The Familiar confirmed: attacker deploys the victim's contract (correct bytecode, CREATE2 address includes bytecode hash), paying gas on the victim's behalf. Victim retries without the create2 step.

**Prior art:** Pashov AI #1 Finding 2, Archethect DT-01, Forefy #2.

**Response — Accepted (no fix needed).** Binding salt to msg.sender would break sender-independent address prediction, complicating cross-EOA and contract-based deployment flows.

---

### [I-02] Multicall msg.value Shared Across Delegatecalls — Duplicate

**Severity:** Informational | **Confidence:** High | **Familiar verdict:** Confirmed (documented design tradeoff)
**Location:** `SafeSummoner.sol` L206-217 — `multicall()`

Each `delegatecall` sub-call sees the same `msg.value`. NatSpec at L203-205 explicitly documents the behavior. Second ETH-consuming sub-call would revert on insufficient balance. Self-contained: only caller's own ETH at risk.

**Prior art:** SCV Scan #2 Finding 1, Archethect DT-02, Forefy #1.

**Response — Accepted (no fix needed).** Standard delegatecall-multicall pattern (Uniswap V3 Router, Seaport). NatSpec documents the behavior. No external attacker can exploit this.

---

### [DC-01] extraCalls Can Override Validated Config — Design Choice

**Severity:** Design Choice | **Confidence:** High | **Familiar verdict:** Confirmed (foot-gun, not vulnerability)
**Location:** `SafeSummoner.sol` L269, L294-296, L835-837

`extraCalls` are appended after validated config calls and execute in the new DAO's `init()` context. A caller could include calls that override SafeSummoner's validated settings. The Familiar confirmed: caller deploys their own DAO, no trust boundary violation. Direct summoning bypasses SafeSummoner entirely. NatSpec labels `extraCalls` as "advanced use."

**Prior art:** Grimoire Moloch audit L-04.

**Response — Accepted (by design).** SafeSummoner is advisory — it provides safe defaults and validation, but does not act as a security boundary. The deployer always has full control over their own DAO's initialization.

---

### [DC-02] Module Parameters Delegated to Singletons — Design Choice

**Severity:** Design Choice | **Confidence:** High | **Familiar verdict:** Confirmed (correct responsibility delegation)
**Location:** `SafeSummoner.sol` — `_buildModuleCalls()`

TapModule parameters (rate, budget, beneficiary) and SeedModule parameters (amounts, deadline, minSupply) are passed through without SafeSummoner validation. Domain-specific constraints are enforced by the singletons' own `configure` functions.

**Response — Accepted (by design).** SafeSummoner validates governance invariants (KF#2, KF#3, KF#11, KF#17). Module-specific domain logic is correctly delegated to singletons. Duplicating validation would create coupling and maintenance burden.
