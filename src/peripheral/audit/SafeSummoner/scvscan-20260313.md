## Phase 4: Confirmed Findings

### 1. msg.value Shared Across Multicall Sub-Calls

**File:** `SafeSummoner.sol` L206-216
**Severity:** Informational

> **Review: Documented design tradeoff.** The NatSpec at L203-205 explicitly warns: "msg.value is shared across all calls — callers sending ETH must ensure only one sub-call consumes it, or that the total is sufficient." This is inherent to any delegatecall-based multicall pattern (same as Uniswap V3 Router, Seaport, etc.) and cannot be "fixed" without breaking the pattern. Self-contained — only the caller's own ETH is at risk, and the caller controls the data array.

**Description:** Each delegatecall sub-call in `multicall()` sees the original `msg.value`. If a caller batches two `safeSummon{value: 1 ether}` calls, the first forwards 1 ETH to SUMMONER, and the second attempts the same — failing if the contract balance is insufficient, or succeeding if residual ETH exists in the contract. No external attacker can exploit this; the risk is solely on the caller misusing their own multicall.

**Code:**
```solidity
function multicall(bytes[] calldata data) public payable returns (bytes[] memory results) {
    results = new bytes[](data.length);
    for (uint256 i; i != data.length; ++i) {
        (bool ok, bytes memory result) = address(this).delegatecall(data[i]);
        // ...
    }
}
```

**Recommendation:** No code change needed. NatSpec warning is sufficient. UIs should ensure only one sub-call in a multicall batch consumes ETH.

**Response — Accepted (no fix needed).** This is standard delegatecall-multicall behavior, identical to Uniswap V3 Router, Seaport, and other production contracts. The NatSpec at L203-205 documents the behavior. The caller controls both the data array and the ETH sent — no external attacker can exploit this. Only the caller's own misconfiguration could cause issues, and they'd simply see a revert. No code change warranted.
