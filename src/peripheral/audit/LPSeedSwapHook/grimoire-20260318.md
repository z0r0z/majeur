## Finding Responses

### [L-01] `quoteExactOut` fee-on-input returns insufficient `amountIn`, causing swap revert

**Severity:** Low | **Confidence:** 90 | **Familiar verdict:** Confirmed
**Location:** `LPSeedSwapHook.sol:666-669` (quote) vs `LPSeedSwapHook.sol:825-844` (swap)

`quoteExactOut` with fee-on-input computes `net = _getAmountIn(amountOut, ...)`, then `daoTax = floor(net * bps / (10000 - bps))`, returning `amountIn = net + daoTax`. When the user passes this exact `amountIn` as `amountInMax` to `swapExactOut`, the swap computes `netMax = floor(amountInMax * (10000 - bps) / 10000)`. Due to floor rounding asymmetry, `netMax` can be 1 less than `net`, causing ZAMM to revert with `InsufficientInputAmount()`.

**Concrete example (Familiar-verified):**
- Reserves: rIn=10000, rOut=10000, poolFee=30 bps, daoFee=300 bps, onInput=true
- User wants amountOut=100
- `_getAmountIn(100, 10000, 10000, 30)` = 102
- Quote: `daoTax = floor(102 * 300 / 9700) = 3`, returns `amountIn = 105`
- Swap: `netMax = floor(105 * 9700 / 10000) = 101`
- ZAMM: `require(102 <= 101)` → **REVERT**

**Root cause:** `floor(a * b / c)` composed with `floor(x * c / (b+c))` is not a round-trip identity — integer floor division can lose 1 unit. The quote formula and swap formula are algebraic inverses in reals but not in integer arithmetic.

**Not a duplicate of KF#6:** KF#6 addressed the *absence* of the `netMax` cap. This finding is about a rounding asymmetry *in* the `netMax` calculation introduced by that patch.

**Impact:** No fund loss — the transaction simply reverts. Any frontend or contract relying on `quoteExactOut` to determine `amountInMax` for fee-on-input swaps will experience intermittent reverts. User must add +1 to the quoted value as a manual buffer.

**Fix:** In `quoteExactOut` L668, ceil the tax instead of flooring:

```diff
- daoTax = (net * bps) / (10_000 - bps);
+ daoTax = (net * bps + (10_000 - bps) - 1) / (10_000 - bps);
```

This guarantees `floor(amountIn * (10000 - bps) / 10000) >= net`.

<summary><strong>Sigil 1 — Reentrancy + CEI Violations</strong></summary>

**Hypothesis:** CEI violations in seed(), swap routing, or configure. Cross-function reentrancy through ETH refunds, ERC20 callbacks, ZAMM callbacks. Transient storage guard bypass.

**Investigation (16 sub-hypotheses):**
- **seed() CEI:** `cfg.seeded = uint40(block.timestamp)` at L295 before all external calls (spendAllowance L341-342, addLiquidity L380, refunds L391-396). Re-entry into seed() hits `cfg.seeded != 0` check at L289.
- **SWAP_LOCK_SLOT:** Transient reentrancy guard at L710-722 protects both `swapExactIn` and `swapExactOut` with the same slot, blocking same-function and cross-function re-entry.
- **SEEDING_SLOT:** Tightly scoped within synchronous ZAMM.addLiquidity call. Only allows beforeAction LP gate bypass, not state-mutating re-entry.
- **receive():** No-op (L914), no exploitable state changes.
- **Cancel during seed:** cancel() checks `cfg.seeded != 0` at L413 — blocked by CEI.
- **Fee config changes mid-seed:** seed() doesn't read fee config, so mid-seed changes via callback are harmless.

**Verdict: DISMISSED** — zero residual reentrancy vectors.
