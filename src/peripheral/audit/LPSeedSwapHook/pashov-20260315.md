## Finding #1 — Permissionless `configure()` Overwrites `poolDAO` Mapping

- **Severity:** High (95 confidence)
- **Validity:** Duplicate of KF#1 (winfunc #3 / zellic #4)

### Description

`configure()` unconditionally writes `poolDAO[poolId] = msg.sender` using a pool ID derived from the token pair and hook address. An attacker could call `configure()` with the same token pair as a victim DAO and overwrite `poolDAO`, causing `beforeAction` to resolve to the attacker's (un-seeded) config and reverting all swaps/LP with `NotReady`.

### Assessment

**Duplicate — already accepted in prior audits.**

`tokenB` is always the DAO's unique shares/loot token (deployed per-DAO by the Summoner). No two DAOs share the same shares token address, so an attacker cannot produce a colliding `poolId` without access to another DAO's shares token. See KF#1 in `SECURITY.md` for full analysis.

## Finding #2 — Fee-on-Output Swap Silently Bypasses User's Slippage Protection

- **Severity:** Medium (80 confidence)
- **Validity:** Valid (patched)

### Description

In `swapExactIn` with fee-on-output, `amountOutMin` was forwarded directly to ZAMM as a gross (pre-tax) floor. The user actually receives `amountOut - tax`, which could be less than their intended `amountOutMin`. A sandwich attacker could drive ZAMM's gross output down to exactly `amountOutMin` while the user nets `amountOutMin * (10_000 - bps) / 10_000`.

### Patch

Pass `0` as ZAMM's minimum and enforce `net >= amountOutMin` after tax deduction:

```diff
- amountOut = ZAMM.swapExactIn{value: msg.value}(
-     poolKey, msg.value, amountOutMin, zeroForOne, address(this), deadline
- );
+ amountOut = ZAMM.swapExactIn{value: msg.value}(
+     poolKey, msg.value, 0, zeroForOne, address(this), deadline
+ );
  uint256 tax = (amountOut * bps) / 10_000;
  uint256 net = amountOut - tax;
+ if (net < amountOutMin) revert Slippage();
```

Note: `swapExactOut` fee-on-output is not affected — the user specifies exact net `amountOut` they want, and the contract computes `gross` correctly. Slippage is on the input side via `amountInMax`.

## Finding #3 — Blacklisted Beneficiary Permanently DoSes All Swaps

- **Severity:** Medium (80 confidence)
- **Validity:** Valid concern, accepted

### Description

When a DAO fee beneficiary is set, every routed swap push-transfers the fee to `ben`. If the beneficiary is blacklisted by the output token (e.g., USDC), every swap reverts.

### Assessment

**Accepted — minimal impact with governance recovery.**

1. The DAO controls the beneficiary via `setBeneficiary()` — governance can change it at any time.
2. Pull-based fee distribution would add significant gas overhead and complexity to every swap.
3. USDC blacklisting of a DAO beneficiary is an extreme edge case requiring external regulatory action.
4. Typical deployment uses DAO shares tokens (not USDC) as the pool's tokenB.

## Finding #4 — Fee-on-Transfer Token Causes Permanent Seed DoS

- **Severity:** Medium (80 confidence)
- **Validity:** Valid concern, accepted (unsupported token type)

### Description

When tokenA or tokenB is a fee-on-transfer token, `seed()` receives fewer tokens than `amtA` from `spendAllowance` but passes the full `amtA` to ZAMM. The refund path then tries to transfer tokens the hook doesn't hold, reverting the entire transaction.

### Assessment

**Accepted — fee-on-transfer tokens are not supported.**

1. `tokenB` is always the DAO's shares/loot token, which does not have transfer fees.
2. `tokenA` is typically ETH (no transfer fee) or a standard ERC20.
3. DAOs configure their own tokens — using a fee-on-transfer token is a misconfiguration.
4. Documented as a design constraint (added to SECURITY.md).

## Finding #5 — Launch Fee Underflow DoSes Swaps When launchBps < feeBps

- **Severity:** Medium (75 confidence)
- **Validity:** Valid (patched)

### Description

If a DAO sets `launchBps` lower than `feeBps` (e.g., via `setFee()` after `setLaunchFee()`), the expression `launch - (launch - target)` in `beforeAction` triggers a Solidity 0.8 underflow revert on every swap during the decay period, DoSing the pool.

### Patch

Made the decay calculation bidirectional — handles both `launch > target` and `launch < target`:

```diff
  if (elapsed < decay) {
-     return launch - (launch - target) * elapsed / decay;
+     if (launch >= target) {
+         return launch - (launch - target) * elapsed / decay;
+     } else {
+         return launch + (target - launch) * elapsed / decay;
+     }
  }
```
