## Round 1 — Defense Verification

**1. msg.sender keying** — *Verified.* All `configure` overloads write to `seeds[msg.sender]`. `cancel()`, `setFee()`, `setLaunchFee()`, `setDaoFee()`, `setBeneficiary()` all read/write via `seeds[msg.sender]` or `daoFees[msg.sender]`. No DAO can modify another DAO's config.

**2. ZAMM-only hook calls** — *Verified.* `beforeAction` L~415: `if (msg.sender != address(ZAMM)) revert Unauthorized()`. Only the hardcoded ZAMM singleton can invoke the hook.

**3. One-shot seeding** — *Verified.* `seed()` checks `if (cfg.seeded != 0) revert AlreadySeeded()` before any external calls. `configure()` also checks `if (seeds[msg.sender].seeded != 0) revert AlreadySeeded()` to block reconfiguration post-seed.

**4. CEI in seed()** — *Verified.* `cfg.seeded = uint40(block.timestamp)` is set immediately after validation, before `spendAllowance`, `addLiquidity`, and refund transfers. Any reentrant call into `seed()` hits `AlreadySeeded`.

**5. Pre-seed LP blocking** — *Verified.* `beforeAction` for non-swap selectors: when `dao != address(0) && seeds[dao].seeded == 0`, checks transient `SEEDING_SLOT`. If not set (external caller), reverts `NotReady`. Only `seed()` sets the flag via `tstore`.

**6. Transient seeding flag** — *Verified.* `tstore(SEEDING_SLOT, address())` immediately before `ZAMM.addLiquidity`, `tstore(SEEDING_SLOT, 0)` immediately after. Scoped to `seed()` only, cleared same transaction.

**7. Condition gates** — *Verified.* `_isReady` checks deadline (`block.timestamp <= cfg.deadline`), ShareSale completion (allowance == 0 or deadline bypass), and minSupply (with snapshot cap for griefing resistance). All three gates traced and correct.

**8. Routing enforcement** — *Verified.* `beforeAction` for swap selectors: `if (daoFees[dao].beneficiary != address(0))` then blocks `IZAMM.swap.selector` entirely via `NotHooked`, and requires `sender == address(this)` for exactIn/exactOut. Direct ZAMM swaps bypass the DAO fee — routing enforcement prevents this.

**9. Transient swap lock** — *Verified.* `lock()` modifier on both `swapExactIn` and `swapExactOut` uses `tload(SWAP_LOCK_SLOT)` / `tstore(SWAP_LOCK_SLOT, address())`, cleared after execution. Prevents reentrancy within routed swaps.

**10. Net slippage check** — *Verified.* `swapExactIn` fee-on-output path: passes `0` to ZAMM as minimum, then checks `if (net < amountOutMin) revert Slippage()` where `net = amountOut - tax`. User's slippage protection applies to actual received amount.

### Invariant Verification

1. **One-shot seed** — *Verified.* `seeded` set before externals; checked at entry of both `seed()` and `configure()`.
2. **DAO-only configuration** — *Verified.* All config functions key to `msg.sender`.
3. **Pre-seed LP blocked** — *Verified.* `poolDAO` reserved at `configure()` time; `beforeAction` blocks non-seed LP on registered-but-unseeded pools.
4. **Post-seed swaps only** — *Verified.* `beforeAction` reverts `NotReady` if `cfg.seeded == 0` for swap selectors; reverts `NotConfigured` if `dao == address(0)`.
5. **Allowance-bounded** — *Verified.* `spendAllowance` is a Moloch-enforced pull; treasury clamping additionally caps to actual balances.
6. **Net slippage** — *Verified.* See defense #10 above.

## Round 2 — Adversarial Hunt

I systematically examined the following attack surfaces and could not produce a surviving finding:

**Refund arithmetic in `swapExactOut` fee-on-input.** Verified algebraically that `amountIn + tax ≤ amountInMax` holds. Since `amountIn ≤ netMax = floor(amountInMax × (10_000 − bps) / 10_000)` and `tax = floor(amountIn × bps / (10_000 − bps))`, the sum `amountIn + tax ≤ amountIn × 10_000 / (10_000 − bps) ≤ amountInMax`. Floor division on tax only tightens the bound. No underflow.

**Stuck ETH via `receive()`.** The contract accepts ETH from anyone (needed for ZAMM refunds and `spendAllowance`). Accidentally sent ETH is irrecoverable but cannot be stolen — swap functions transfer specific computed amounts, never `address(this).balance`. Matches common singleton patterns; no fund-loss risk to users.

**Standing max approvals via `ensureApproval`.** After any routed swap, the contract retains `type(uint256).max` approval to ZAMM for that token. Since the contract should hold zero token balance between transactions (all paths fully distribute), and ZAMM is the trusted singleton, this is inert. Cross-checked: no code path leaves tokens in the contract post-execution.

**Cross-function reentrancy (swap → seed).** `seed()` lacks the `lock` modifier but has CEI — `cfg.seeded` is set before any external call. Reentrant `seed()` hits `AlreadySeeded`. Reentrant `configure()` also hits `AlreadySeeded`. Safe.

**Stale `poolDAO` via concurrent configure.** Two DAOs configuring the same token pair could overwrite each other's `poolDAO` entry. But `tokenB` is always the DAO's unique shares token (deployed per-DAO by Summoner), making token pair collision infeasible. This is the same reasoning that accepted KF#1.

**Quote-to-swap timestamp drift.** `effectiveFee` depends on `block.timestamp`, so launch decay advances between quote and execution. This is inherent to any time-decaying parameter and mitigable by user slippage settings. Not a bug.

**Direction-specific zero bps with active beneficiary.** `setDaoFee` requires `(buyBps | sellBps) != 0` when beneficiary is set, but one direction can be zero. When `bps == 0` for a direction, `tax = 0`, `net = amountOut`, swap passes through correctly. No edge-case failure.

All 10 defense mechanisms verified. All 6 invariants hold. No new findings survive deduplication against the 7 known findings + 6 dismissed findings + 6 design choices. The contract is well-hardened after 5 audit rounds and the corresponding patches.
