### Surviving finding

## [MEDIUM-1] Stale `poolDAO` claims survive reconfiguration and can misregister legacy pairs

**Severity:** Medium
**Confidence:** 93
**Location:** `src/peripheral/LPSeedSwapHook.sol:248-278, 295, 360-366, 410-423, 510-539`

**Description**

`configure()` overwrites `seeds[msg.sender]` in place but does **not** remove the caller’s previously reserved `poolDAO` entry before reserving the new pool (`250-278`). `cancel()` only deletes the **current** pool derived from the current `seeds[msg.sender]` (`415-423`), so older reservations are orphaned.

Later, `beforeAction()` authorizes swaps by checking only:

* `dao = poolDAO[poolId]` (`510`)
* `seeds[dao].seeded != 0` (`529-531`)

It does **not** verify that:

* this `poolId` equals the DAO’s current configured pool, or
* this exact pool was ever seeded.

That makes stale reservations security-relevant.

**Attack path**

1. DAO configures pair **P1**.
   `poolDAO[P1] = dao` (`267-278`).

2. Before seeding, DAO configures pair **P2**.
   `seeds[dao]` is overwritten, and `poolDAO[P2] = dao`, but `poolDAO[P1]` is left behind (`250-278`).

3. DAO later seeds **P2**.
   `seeds[dao].seeded` becomes nonzero (`295`, `360-366`).

4. Now any third party can create/use pool **P1** with this hook.
   In `beforeAction()`, **P1** resolves to `dao` through the stale `poolDAO[P1]`, and because `seeds[dao].seeded != 0`, swaps on **P1** are treated as registered/seeded (`510`, `529-539`) even though **P1** was never seeded by the DAO.

Impacts:

* A stale, never-seeded pair can become permanently associated with the DAO’s hook identity.
* If DAO routing is enabled, that stale pair inherits the DAO’s routing/fee rules.
* If the DAO cancels after reconfiguration instead of seeding, the stale old pair remains blocked until some later `configure()` overwrites it.

**Disproof attempt**

* **Not KF#1:** no pool-ID collision is needed; this occurs with the DAO’s own historical pool IDs.
* **Not KF#2:** this is not the pre-reservation window; it is about **failure to clean up old reservations** after reconfiguration.
* **Not a design choice:** repeated `configure()` is explicitly allowed pre-seed (`248` only blocks already-seeded DAOs), but there is no compensating cleanup.
* **Can `cancel()` fix it?** No. `cancel()` hashes only the DAO’s **current** config (`415-420`), so older pool IDs are not deleted.
* **Can later callers overwrite the stale mapping?** Only while `seeds[existing].seeded == 0` (`275`). Once the DAO seeds some later pair, the stale mapping becomes sticky against third-party overwrite.

**Recommendation**

Track and clean up reservations explicitly.

A robust fix is:

1. Store the DAO’s currently reserved `poolId` in state.
2. On every pre-seed `configure()`, if an old reservation exists, delete it before writing the new one.
3. On `cancel()`, delete that stored reserved ID directly.
4. For swaps, store a **seededPoolId** separately and require `poolId == seededPoolId` in `beforeAction()` instead of trusting any historical `poolDAO` entry.

A lighter fix is to delete the prior pool ID before overwriting `seeds[msg.sender]`, but a dedicated `reservedPoolId/seededPoolId` split is cleaner and closes both the DoS and false-registration angles.

### Resolution — FALSE POSITIVE (2026-03-18)

The contract already implements the "lighter fix" at `configure()` lines 250-260: when a DAO reconfigures with a different token pair, the old `poolDAO` entry is explicitly deleted before the new one is written. `cancel()` (line 432) also deletes the current entry. The described attack path fails at step 2 — the stale `poolDAO[P1]` is cleaned up, so step 4 would revert `NotConfigured()`.

Same-pair reconfigurations are also safe: the pool ID is deterministic from the token pair, so the same `poolDAO[poolId]` slot is simply overwritten.

This finding was likely generated against a prior revision of the code before the cleanup block was added.
