## Finding #4 ‚Äî poolDAO mapping can be overwritten

- **Severity:** High
- **Validity:** Valid concern (Duplicate), impractical in deployment

### Description

Both `configure` (setting `poolDAO[poolId] = msg.sender`) and `seed` (setting `poolDAO[poolId] = dao`) update the mapping without authorization or overwrite guards. Since `poolId` is deterministic from the token pair and hook address, any caller can recompute an existing `poolId` and replace the DAO association, potentially freezing swaps or applying attacker-chosen fees.

### Assessment

**Valid as a general concern but impractical in deployment. Duplicate of winfunc #3.**

The `poolId` is derived from `PoolKey{id0: 0, id1: 0, token0, token1, feeOrHook}`. In practice:

1. **`tokenB` is always the DAO's shares/loot token** ‚Äî a unique contract per DAO deployed by the Summoner. No two DAOs share the same shares token address.
2. **Pool collision requires identical token pairs** ‚Äî since `tokenB` is unique per DAO, an attacker cannot produce a `poolId` that collides with another DAO's pool.
3. **`configure` is called by the DAO itself** during init ‚Äî `msg.sender` is the DAO. An attacker calling `configure` with different tokens gets a different `poolId`.

The Zellic PoC uses the same `sharesAddr` for the attacker's `configure` call, which would require the attacker to know and use another DAO's shares token. But the attacker's `configure` call sets `poolDAO[poolId] = attacker_address`, which would work ‚Äî except the attacker cannot call `configure` with another DAO's shares token because `configure` is typically called within the DAO's init transaction.

**Zellic marked this "Unfixable."** The practical mitigation is that the pool key space is DAO-namespaced by the unique shares token. **Accepted as theoretical concern.**

## Finding #9 ‚Äî Global seeding flag enables cross-pool bypass

- **Severity:** High
- **Validity:** Invalid

### Description

Zellic reports that the global transient `SEEDING_SLOT` flag, set during `seed()`, could be exploited by a callback-capable token to reenter ZAMM and perform actions on a different pool while the bypass is active.

### Assessment

**Invalid.** The attack requires a callback-capable token used during seeding, but:

1. **Tokens are DAO-configured** ‚Äî the DAO sets `tokenA` and `tokenB` in `configure()`. These are set by the DAO itself during initialization, not by an attacker.
2. **`ZAMM.addLiquidity` calls a trusted singleton** ‚Äî ZAMM is a constant singleton address (`0x000000000000040470635EB91b7CE4D132D616eD`). The token transfers within `addLiquidity` are standard `transferFrom` calls.
3. **Cross-pool collision impossible** ‚Äî as established in Finding #4's assessment, different DAOs produce different `poolId`s due to unique shares tokens.
4. **Transient storage scope** ‚Äî the `SEEDING_SLOT` flag is cleared immediately after `addLiquidity` returns (line 275). Even if a callback occurred during the call, the flag is correctly bounded.

**No fix needed.**

## Finding #5 ‚Äî Inverted minSupply readiness gate

- **Severity:** Medium
- **Validity:** Invalid

### Description

Zellic reports that `_isReady` treats `minSupply` as an upper bound instead of a minimum threshold, claiming the comparison `if (daoBal > cfg.minSupply) return false` is inverted.

### Assessment

**Invalid.** The audit misinterprets the semantics of `minSupply`.

The NatSpec at L159 explicitly states: `uint128 minSupply; // if set, seed only after DAO's tokenB balance <= minSupply`. The comment at L392 confirms: "Supply gate: DAO's tokenB balance must be at or below threshold."

The intended behavior: The DAO starts with a large `tokenB` (shares) balance, sells shares via ShareSale, and seeding should only occur after the remaining supply drops to/below `minSupply` ‚Äî i.e., enough shares have been sold. The check `daoBal > cfg.minSupply ‚Üí not ready` correctly implements this ceiling logic.

The variable name `minSupply` means "minimum remaining supply threshold to trigger seeding" ‚Äî it's the supply floor at which the DAO is considered ready to seed LP, not a minimum balance requirement.

The Zellic PoC's own validation output confirms the patch was applied and reverts with `InvalidParams()` instead of `NotReady()`, demonstrating the original code was already correct.

**No fix needed.**

## Finding #6 ‚Äî DAO balance check enables seeding DoS

- **Severity:** Medium
- **Validity:** Valid (Duplicate)

### Description

Since `_isReady` uses `balanceOf` to check `minSupply`, any holder can donate `tokenB` to the DAO to push the balance above the threshold, causing `seed()` to revert indefinitely.

### Assessment

**Valid, accepted (minimal impact). Duplicate of winfunc #24 (minSupply dusting griefing).**

This is a standard "donation griefing" pattern. Mitigating factors:

1. **Cost to attacker:** The attacker must permanently transfer their own tokens to the DAO. These tokens become DAO treasury assets ‚Äî the attacker loses value.
2. **DAO can reconfigure:** The DAO can call `cancel()` and re-`configure()` with an updated `minSupply` threshold accounting for the donated tokens.
3. **Alternative: remove minSupply gate** ‚Äî the DAO can reconfigure with `minSupply = 0` to bypass the gate entirely.
4. **Transient griefing only** ‚Äî the DAO retains full control to recover via governance.

**Not applied ‚Äî accepted as low-impact griefing with governance recovery.**
