# Reentrancy during readiness check
**#3**
- Severity: Critical
- Validity: Invalid

## Targets
- seed (LPSeedSwapHook)

## Affected Locations
- **LPSeedSwapHook.seed**: Single finding location

## Description

The `seed` function calls `_isReady` before it marks the seed as completed. `_isReady` performs several external calls (`IShareSale.sales`, `IMoloch.allowance`, and token `balanceOf`), any of which can execute attacker-controlled code. A malicious external contract can reenter `seed` during this readiness check while `cfg.seeded` is still zero, so the reentrant call passes the `AlreadySeeded` guard and performs the full liquidity seeding. After the reentrant call finishes, the original call continues and repeats `spendAllowance` and `addLiquidity`, effectively doubleтАСspending the DAOтАЩs allowance within a single transaction.

## Root cause

External calls inside `_isReady` occur before `cfg.seeded` is updated, leaving a reentrancy window with stale seed state.

## Impact

An attacker controlling the shareтАСsale contract or one of the tokens can force multiple seeding operations, consuming the DAOтАЩs allowances more than intended and locking extra assets into the pool. With large or unlimited allowances, this can drain a significant portion of the DAOтАЩs treasury in a single transaction.

## Remediation

**Status:** Incomplete

### Explanation

Apply checksтАСeffectsтАСinteractions in `seed`: mark `cfg.seeded` (and any other seed state) or activate a reentrancy guard before calling `_isReady`/any external token or sale contract, then perform the external calls only after the state is locked so reentrancy cannot trigger another seeding.

# Stale configs can hijack or de-register pools
**#2**
- Severity: High
- Validity: Unreviewed

## Targets
- configure (LPSeedSwapHook)
- cancel (LPSeedSwapHook)
- beforeAction (LPSeedSwapHook)
- seed (LPSeedSwapHook)

## Affected Locations
- **LPSeedSwapHook.configure**: `configure` assigns `poolDAO[poolId] = msg.sender` while only preventing overwrites when the prior DAO is marked as seeded, so ownership is not locked during the configured-but-unseeded phase. Tightening this logic (e.g., disallowing overwrites once configured, or requiring explicit owner-controlled replacement/expiry) prevents pre-seed hijacking and eliminates the stale-config window used by later `cancel` attacks.
- **LPSeedSwapHook.cancel**: `cancel` deletes `poolDAO[poolId]` based on the callerтАЩs stored configuration without checking that `poolDAO[poolId]` still equals the caller, allowing stale configurators to erase an active poolтАЩs mapping. Requiring `poolDAO[poolId] == msg.sender` (and/or that the pool is still unseeded) before deletion prevents deregistering pools seeded by others.
- **LPSeedSwapHook.beforeAction**: `beforeAction` relies on `poolDAO[poolId]` being non-zero and reverts as `NotConfigured` when it is cleared, so deleting `poolDAO` directly translates into a swap-blocking denial of service. This is where the unauthorized mapping deletion materializes as halted trading and disrupted fee routing.
- **LPSeedSwapHook.seed**: `seed` writes/overwrites `poolDAO[poolId]` to the seeding DAO, but stale configurations created earlier remain able to affect the same `poolId` later via `cancel`. The seeded state and mapping update here are the state that a stale `cancel` later invalidates, so the lifecycle needs consistent authority checks across both functions.

## Description

The pool-to-DAO association stored in `poolDAO` is not treated as a strictly owned, single-writer record across the pool lifecycle. In `configure`, a caller can overwrite `poolDAO[poolId]` as long as the previously recorded DAO has not yet seeded, enabling тАЬlast writer winsтАЭ ownership hijacking before seeding. Separately, `cancel` can delete `poolDAO[poolId]` based only on the callerтАЩs own (possibly stale) configuration, without confirming that the caller is still the currently recorded DAO for that pool. This allows an attacker to preconfigure a pool, wait for a legitimate DAO to seed and go live, and then use `cancel` to wipe the active `poolDAO` entry. Once cleared, hook logic that expects `poolDAO` to be set will treat the pool as unconfigured and revert, while the legitimate DAO may be unable to restore the mapping due to seeding/guard conditions.

## Root cause

The contract fails to enforce stable ownership of `poolDAO[poolId]` (single-authority updates) and lets `cancel` mutate `poolDAO` without verifying the current mapping owner (`poolDAO[poolId] == msg.sender`).

## Impact

An attacker can front-run or later overwrite an unseeded poolтАЩs DAO association, capturing future fee/tax routing and potentially blocking the intended DAO from seeding. After a victim seeds, the attacker can still wipe the active mapping via a stale `cancel`, causing swaps to revert and effectively freezing trading through the hook for that pool. The attacker may then reconfigure/reseed to take over the poolтАЩs DAO association and divert future fees.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.30;

import {Test} from "forge-std/Test.sol";
import {Moloch} from "src/Moloch.sol";
import {SafeSummoner, Call} from "src/peripheral/SafeSummoner.sol";
import {LPSeedSwapHook, IZAMM} from "src/peripheral/LPSeedSwapHook.sol";

contract LPSeedSwapHookStaleConfigPoC is Test {
    SafeSummoner internal safe;
    LPSeedSwapHook internal lpSeed;

    address internal alice = address(0xA11CE);
    address internal attacker = address(0xBEEF);

    address internal constant ZAMM_ADDR = 0x000000000000040470635EB91b7CE4D132D616eD;

    function setUp() public {
        vm.createSelectFork("main");
        vm.deal(alice, 100 ether);
        vm.deal(attacker, 100 ether);
        safe = new SafeSummoner();
        lpSeed = new LPSeedSwapHook();
    }

    function _deployWithSeed(bytes32 salt, uint128 amtA, uint128 amtB)
        internal
        returns (address dao, address sharesAddr)
    {
        address[] memory h = new address[](1);
        h[0] = alice;
        uint256[] memory s = new uint256[](1);
        s[0] = 100e18;

        dao = safe.predictDAO(salt, h, s);
        sharesAddr = safe.predictShares(dao);

        Call[] memory extra = new Call[](3);
        extra[0] =
            Call(dao, 0, abi.encodeCall(Moloch.setAllowance, (address(lpSeed), address(0), amtA)));
        extra[1] = Call(dao, 0, abi.encodeCall(Moloch.setAllowance, (address(lpSeed), dao, amtB)));
        extra[2] = Call(
            address(lpSeed),
            0,
            abi.encodeWithSignature(
                "configure(address,uint128,address,uint128,uint16,uint40,address,uint128,address,address)",
                address(0),
                amtA,
                sharesAddr,
                amtB,
                uint16(0),
                uint40(0),
                address(0),
                uint128(0),
                address(0),
                dao
            )
        );

        SafeSummoner.SafeConfig memory c;
        c.proposalThreshold = 1e18;
        c.proposalTTL = 7 days;

        address deployed = safe.safeSummon(
            "SeedDAO", "SEED", "", 1000, true, address(0), salt, h, s, new uint256[](0), c, extra
        );
        assertEq(deployed, dao);

        vm.deal(dao, uint256(amtA) + 10 ether);
    }

    function test_StaleCancelClearsPoolDAO() public {
        (address dao, address sharesAddr) = _deployWithSeed(bytes32(uint256(1)), 1e18, 1000e18);

        // Attacker configures the same pool after deployment but before seeding.
        vm.prank(attacker);
        lpSeed.configure(address(0), 1e18, sharesAddr, 1000e18, 0, address(0), 0);

        (, uint256 poolId) = lpSeed.poolKeyOf(dao);
        assertEq(lpSeed.poolDAO(poolId), attacker);

        // Legitimate DAO seeds and poolDAO points back to it.
        lpSeed.seed(dao);
        assertEq(lpSeed.poolDAO(poolId), dao);

        // Swaps are allowed while poolDAO is set.
        vm.prank(ZAMM_ADDR);
        uint256 feeBps = lpSeed.beforeAction(IZAMM.swapExactIn.selector, poolId, address(0x1234), "");
        assertEq(feeBps, 30);

        // Attacker cancels using stale config and wipes poolDAO.
        vm.prank(attacker);
        lpSeed.cancel();
        assertEq(lpSeed.poolDAO(poolId), address(0));

        // Now swaps revert as unconfigured, freezing the pool.
        vm.prank(ZAMM_ADDR);
        vm.expectRevert(LPSeedSwapHook.NotConfigured.selector);
        lpSeed.beforeAction(IZAMM.swapExactIn.selector, poolId, address(0x1234), "");

        // Legitimate DAO cannot restore the mapping after seeding.
        vm.prank(dao);
        vm.expectRevert(LPSeedSwapHook.AlreadySeeded.selector);
        lpSeed.configure(address(0), 2e18, sharesAddr, 2000e18, 0, address(0), 0);
    }
}
```

## Remediation

**Status:** Complete

### Explanation

Enforce single-writer ownership of `poolDAO` by rejecting `configure` calls when a pool already has a different owner, and prevent stale `cancel` calls from clearing mappings by requiring `poolDAO[poolId]` to match `msg.sender` before deletion.

# Minimum supply check inverted
**#5**
- Severity: High
- Validity: Invalid

## Targets
- seedable (LPSeedSwapHook)

## Affected Locations
- **LPSeedSwapHook.seedable**: Single finding location

## Description

The readiness gate in `seedable` relies entirely on `_isReady`, and the minimum-supply branch in `_isReady` uses a `>` comparison that treats balances above `minSupply` as not ready. Given the variable name and contract summary, `minSupply` is meant to be a minimum threshold before seeding, but the code instead requires the DAOтАЩs `tokenB` balance to be at or below that threshold. This flips the intended gating logic: seeding becomes possible while the DAO has less than the required balance, and it becomes impossible once the balance exceeds the threshold. Because `seedable` is the canonical readiness check, any external caller that can trigger seeding can bypass the minimum-supply constraint, or a third party can block seeding by transferring `tokenB` to push the DAO balance above `minSupply`. The result is premature seeding or a denial of seeding that contradicts the intended launch conditions.

## Root cause

The minimum-supply condition in `_isReady` uses `if (daoBal > cfg.minSupply) return false;`, which inverts the intended minimum-threshold check.

## Impact

Seeding can occur before the DAO reaches the intended minimum balance, undermining launch sequencing and possibly enabling mispriced initial liquidity. Conversely, an attacker can donate `tokenB` to the DAO to exceed `minSupply`, causing `seedable` to return false and preventing seeding indefinitely. This breaks the governance-configured readiness guard and can stall or mis-time the pool launch.

## Remediation

**Status:** Incomplete

### Explanation

Invert the minimum-supply guard in `_isReady` so it blocks seeding only when the DAO balance is **below** `minSupply` (e.g., `if (daoBal < cfg.minSupply) return false;`), ensuring readiness becomes true once the threshold is met and cannot be disabled by exceeding it.

# Global seeding flag bypasses pool readiness
**#6**
- Severity: High
- Validity: Invalid

## Targets
- seed (LPSeedSwapHook)

## Affected Locations
- **LPSeedSwapHook.seed**: Single finding location

## Description

The hook uses `SEEDING_SLOT` as a single transient тАЬseeding in progressтАЭ flag that is set in `seed` and read in `beforeAction` to bypass the `NotReady` gate for non-swap actions. This flag is not tied to a specific pool or DAO, so any pool can be treated as authorized while any seed call is in progress. During the `seed` flow, external calls (token transfers inside `ZAMM.addLiquidity` or callbacks from the DAO/share-sale) can reenter and invoke ZAMM actions for a different pool. When the DAO later seeds and swaps open, the attacker can arbitrage the manipulated price, extracting value from the DAOтАЩs liquidity or forcing launch at a hostile price.

## Root cause

`SEEDING_SLOT` is a global transient flag and `beforeAction` does not validate that the active seeding context matches the `poolId` being operated on. This allows any poolтАЩs readiness checks to be bypassed during any seed transaction.

## Impact

An attacker can bypass the readiness gate and add initial liquidity to any unseeded DAO pool during another DAOтАЩs seed, setting an arbitrary initial price. Once the DAO later seeds and swaps are enabled, the attacker can arbitrage the mispriced pool to extract value from the DAOтАЩs liquidity or force the pool to launch at an unintended price.

**Status:** Incomplete

Store the active seeding pool identifier (or a perтАСpool seeding flag) when `seed` is called, and in `beforeAction` require that the pool being operated on matches that active seeding pool; otherwise enforce normal readiness checks. # ETH ignored when configured as tokenB
**#4**
- Severity: Medium
- Validity: Invalid

## Targets
- seed (LPSeedSwapHook)

## Affected Locations
- **LPSeedSwapHook.seed**: Single finding location

## Description

The function treats either `tokenA` or `tokenB` as native ETH in the balance capping logic, implying ETH is supported in either slot. However, the ETH value sent to `ZAMM.addLiquidity` is computed solely as `ethValue = tokenA == address(0) ? amtA : 0`, which ignores the case where `tokenB` is ETH. After token sorting, `amt0` can represent the ETH amount even when `tokenA` was an ERC20, but `msg.value` will still be zero. This underfunds the call and causes `addLiquidity` to revert, blocking seeding for those pairs.

## Root cause

`ethValue` is derived only from `tokenA` before token ordering, so native ETH in `tokenB` is never forwarded to `addLiquidity`.

## Impact

Pools where ETH is configured as `tokenB` cannot be seeded, resulting in a denial of service for those configurations. If the DAO cannot reconfigure or the deadline has passed, the pool launch may be permanently blocked.

## Remediation

**Status:** Incomplete

### Explanation

Derive the ETH value after token ordering (or check both tokenA and tokenB for the native token) and forward that amount in the addLiquidity call so msg.value is set whenever either side is ETH, ensuring pools with ETH as tokenB can be seeded.
