## Finding #1 — Arb-Protection Clamping Uses Hardcoded 1e18 Scaling

- **Severity:** High (85 confidence)
- **Validity:** False positive

### Description

The scanner flagged `maxShares = payAmt * 1e18 / salePrice` (line 326) as assuming
an 18-decimal pay token. It proposed normalizing `payAmt` by the pay token's actual
decimals before the division.

### Assessment

**False positive — the `1e18` is the shares scaling factor, not a decimal assumption.**

Both `ShareSale` and `BondingCurveSale` define `salePrice` as "payToken-wei per 1e18
share-wei" and compute cost as `cost = amount * price / 1e18`. Line 326 is the
algebraic inverse:

```
cost   = shares * salePrice / 1e18     (buy direction)
shares = payAmt * 1e18      / salePrice (seed arb-clamp)
```

For USDC (6 decimals) at $10/share: `salePrice = 10e6`, `payAmt = 10e6` →
`maxShares = 10e6 * 1e18 / 10e6 = 1e18` (one whole share). Correct.

The convention is enforced by the sale contracts themselves — operators express price
in native payToken-wei, and the 1e18 factor cancels consistently. No patch needed.

## Finding #2 — Blacklisted Beneficiary DoS on Routed Swaps

- **Severity:** Low (65 confidence)
- **Validity:** Duplicate of pashov-20260315 #3

### Description

If a DAO fee beneficiary is blacklisted by the fee token (e.g. USDC), all swaps
through the pool revert until governance updates the beneficiary.

### Assessment

**Duplicate — already accepted in pashov-20260315 #3.** Governance can change
the beneficiary at any time via `setBeneficiary()`. Accepted risk.
