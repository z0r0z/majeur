## Invalidated Findings

### ~~ETH Permanently Trapped When Token Cap Limits ZAMM Liquidity Seeding~~

**Invalidated.** ZAMM `addLiquidity` when `supply == 0` (first pool creation) sets `(amount0, amount1) = (amount0Desired, amount1Desired)` — it consumes exactly `ethForLP` with no leftover (ZAMM.sol:528-529). The `beforeAction` hook blocks pool creation before graduation via the transient `SEEDING_SLOT` bypass, guaranteeing `supply == 0`. Since `msg.value == amount0`, ZAMM's refund path (ZAMM.sol:549-552) is never entered. The `tokensForLP` cap on line 920 only affects the pool ratio, not ETH consumption.

**Response:** Agreed — invalidation confirmed. ZAMM first-pool semantics consume both desired amounts exactly; no ETH is trapped.

### ~~Max-Allowance Approve on Non-Standard ERC20 Permanently Traps Raised ETH~~

**Downgraded to informational.** Only affects the `configure()` path — `launch()` clones are explicitly exempt (line 934 comment: "no-op for launch() clones (ZAMM is allowance-exempt)"). Tokens like UNI/COMP that revert on `approve(type(uint256).max)` would block graduation, but `configure()` is a power-user path where the caller assumes responsibility for token compatibility.

**Response:** Agreed — downgrade to informational accepted. `configure()` callers assume token compatibility responsibility; `launch()` clones are exempt.

### ~~`safeTransfer` Assembly Overwrites Free Memory Pointer to Zero~~

**Invalidated.** This is the standard Solady `safeTransfer` pattern. The `mstore(0x34, 0)` cleanup is intentional — Solidity's codegen for `emit` uses inline `log` opcodes with scratch-space ABI encoding, not the free memory allocator. The `memory-safe` annotation is correct. Battle-tested across thousands of deployed contracts.

**Response:** Agreed — invalidation confirmed. Standard Solady pattern, battle-tested.

### Below-threshold details

[75] **1. Creator Can DoS All Fee-Bearing Trades by Transferring Role to Non-Payable Contract**

`ClassicalCurveSale.setCreator` · Confidence: 75

**Description**
`setCreator` allows the current creator to transfer the role to a contract that reverts on ETH receipt; once set, every `buy()`, `sell()`, `buyExactIn()`, and `sellExactOut()` call that incurs a non-zero fee will revert at `safeTransferETH(creator, fee)`, permanently bricking trading for all users of that token's curve. Acknowledged in NatSpec (line 1008).

**Response:** Duplicate of Known Finding #3. Accepted — self-inflicted by creator against their own token's buyers. Acknowledged in NatSpec.

[75] **2. Fee-on-Transfer Token Accounting Mismatch via `configure()`**

`ClassicalCurveSale.configure` · Confidence: 75

**Description**
`configure()` records `cap` and `lpTokens` as available after `safeTransferFrom(token, address(this), cap + lpTokens)`, but for a fee-on-transfer ERC20 the contract receives fewer tokens than recorded, causing `buy()` transfers and `graduate()` burns to revert on insufficient balance. `configure()` path only — `launch()` clones are immune.

**Response:** Duplicate of Known Finding #2. Accepted — `configure()` is a power-user path where the caller assumes token compatibility responsibility. `launch()` clones are immune.

[75] **3. Rebasing Token Causes Permanent Graduation DoS**

`ClassicalCurveSale.graduate` · Confidence: 75

**Description**
When a rebasing token registered via `configure()` rebases downward, the contract's actual balance falls below the tracked `cap - sold`, causing `graduate()` to revert when it attempts `safeTransfer(token, address(0xdead), unsold)` and permanently blocking graduation. `configure()` path only.

**Response:** Variant of Known Finding #2. Accepted — same root cause (nonstandard token via `configure()` path). `launch()` clones are immune.
