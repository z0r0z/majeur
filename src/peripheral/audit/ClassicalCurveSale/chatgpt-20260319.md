### Round 1 summary

I verified **6/8 defenses** and **5/7 invariants**.

The two substantive breaks are:

1. **Unchecked downcasts** undermine the stated overflow/target defenses.
2. **Pre-seed LP gating is not airtight** because pool registration happens before the hook’s transient bypass becomes relevant.

**Team response to Round 1:**

1. **Overflow guards "Violated"** — Valid. Our initial dismissal was wrong — Solidity 0.8 explicit narrowing casts (`uint128(x)`) silently truncate, they do NOT revert. Added explicit `type(uint128).max` bounds checks for all downcasted fields. **Fixed.**
2. **Graduation target validation "Violated"** — Valid. Same root cause as above — `uint128(graduationTarget)` could truncate the validated value. Now bounds-checked before storage. **Fixed.**
3. **I-1 "Violated"** — Accepted rewording. Invariant restated: "During the active bonding-curve phase, before `graduate()` executes, `raisedETH` tracks net ETH from curve trading." Post-graduation, `raisedETH` is consumed and no longer a live invariant.
4. **I-7 "Violated"** — Valid. Fixed by moving `poolToken[poolId] = token` after `ensureApproval()`. See MEDIUM-1 response.

### [MEDIUM-1] Pre-seed pool can be manipulated during `graduate()` via token reentrancy

* **Severity:** Medium
* **Confidence:** 88
* **Location:** `ClassicalCurveSale.sol:748-758`, `783-805`, `1294-1313`

**Description**

`graduate()` registers the pool with `poolToken[poolId] = token` before calling `ensureApproval(token, address(ZAMM))` (`748-752`). `beforeAction()` treats any non-swap call as allowed whenever `poolToken[poolId] != 0` and only falls back to the transient seeding check when `poolToken[poolId] == 0` (`792-805`).

That means the hook’s “only `graduate()` may create the initial LP” protection is bypassable during the reentrancy window between pool registration and the official `ZAMM.addLiquidity()` call.

Because `ensureApproval()` can invoke an arbitrary token’s `approve()` (`1294-1313`), a malicious token configured through `configure()` can reenter during `approve()` and call ZAMM to add the **first** liquidity at an attacker-chosen ratio.

That breaks the stated assumption behind seeding with `amount0Min = amount1Min = 0`.

**Attack path**

1. Attacker deploys a malicious ERC20 and configures the sale through `configure()`.
2. The curve reaches graduation.
3. `graduate()` runs and writes `poolToken[poolId] = token` (`750`).
4. `ensureApproval()` calls the token’s `approve()` (`1301-1306`).
5. The token reenters and calls `ZAMM.addLiquidity()` for the same pool before the sale contract’s own `addLiquidity()`.
6. `beforeAction()` allows that LP operation because `poolToken[poolId] != 0` (`798-805` is skipped).
7. The attacker sets the initial pool ratio; the sale contract’s later `addLiquidity(..., 0, 0, ...)` is forced to seed against the attacker’s ratio.

**Disproof attempt**

This does **not** affect the `launch()` path’s built-in ERC20 clone; that clone has no callback path in `approve()`. The exploit needs the `configure()` path with a nonstandard/malicious token. But `configure()` is explicitly part of the supported surface, so the issue is real for that path.

**Recommendation**

Move `ensureApproval()` before `poolToken[poolId] = token`, and register the pool immediately before the ZAMM call. Also tighten `beforeAction()` so non-swap LP ops are only allowed when a dedicated “seeding in progress” condition is true, not merely when `poolToken[poolId]` is nonzero.

**Response:** Valid finding. `poolToken[poolId] = token` moved after `ensureApproval()` so during the `approve()` callback the pool is still unregistered and `beforeAction()` blocks LP operations. The transient seeding bypass is also not set yet at that point, fully closing the reentrancy window. Only affects the `configure()` path with malicious tokens — the `launch()` path's ERC20 clone has no callback surface. **Status: Fixed**

### [MEDIUM-2] Creator can change LP recipient after graduation trigger and before seeding

* **Severity:** Medium
* **Confidence:** 85
* **Location:** `ClassicalCurveSale.sol:838-843`, `1065-1076`, `754-761`, `1105-1113`

**Description**

`setLpRecipient()` only blocks changes after `c.seeded` is true (`838-843`). It does **not** block changes once the sale is already frozen with `c.graduated = true`.

So after buyers push the curve over its graduation threshold, the creator can still change the future LP recipient and then call `graduate()`. Since `graduate()` uses the live `c.lpRecipient` at seeding time (`754`), the creator can replace a previously advertised burn/lock address with their own address at the last moment.

`multicall()` makes this atomic (`1105-1113`).

**Attack path**

1. Creator launches/configures the sale with `lpRecipient` set to a burn or lock address.
2. Buyers purchase until `_checkGraduation()` sets `c.graduated = true` (`1072-1075`).
3. Before anyone else seeds LP, the creator calls `multicall([setLpRecipient(token, creator), graduate(token)])`.
4. `graduate()` mints LP to the new recipient (`754-761`).
5. The creator now controls LP that buyers may have believed would be burned/locked.

**Disproof attempt**

Permissionless `graduate()` narrows the window, but does not remove it. The creator can still frontrun the seed transaction, or bundle the change atomically with `graduate()` via `multicall()`. If the threshold-crossing buy pays a creator fee, a contract creator can even react inside that same transaction via its ETH-receiving fallback.

**Recommendation**

Freeze `lpRecipient` once `c.graduated` becomes true, or make it immutable after configuration.

**Response:** Valid finding. `setLpRecipient()` now checks `c.graduated` instead of `c.seeded`, freezing the LP recipient as soon as graduation triggers. The creator can no longer swap a burn address for their own between graduation and seeding. **Status: Fixed**

### [LOW-3] Unchecked downcasts let configured runtime values differ from validated/emitted values

* **Severity:** Low
* **Confidence:** 91
* **Location:** `ClassicalCurveSale.sol:316-343`, especially `336-341`, plus `369`

**Description**

`_configure()` validates and emits 256-bit configuration inputs, but persists several of them as unchecked `uint128(...)` values:

* `startPrice` (`336`)
* `endPrice` (`337`)
* `graduationTarget` (`339`)
* `lpTokens` (`341`)

The input `graduationTarget` is validated against `maxETH` (`316-325`), but the **stored** target can be a truncated lower value. The same applies to price endpoints and LP token reserve.

So there are parameter sets where the on-chain transaction/event says one thing, but the live runtime configuration stored in `_curves[token]` is different.

**Attack path**

1. Attacker calls `launch()` or `configure()` with oversized values that pass the input-time checks.
2. `_configure()` emits `Configured(...)` with the original values (`369`).
3. Storage keeps truncated values (`336-341`).
4. Trading and graduation later use the truncated values, not the emitted ones.

That creates a bait-and-switch surface for off-chain systems that trust emitted config or calldata more than live storage.

**Disproof attempt**

This is bounded in scope:

* `cap` is largely constrained indirectly by the `virtualReserve` check (`303-315`),
* some extreme values still fail `maxETH` or token-supply constraints.

But the bug remains for the downcasted fields above.

**Recommendation**

Add explicit `type(uint128).max` bounds checks for every downcasted field before writing storage.

**Response:** Valid finding — our initial dismissal was incorrect. Solidity 0.8 checked arithmetic does NOT extend to explicit narrowing casts; `uint128(x)` silently truncates high bits without reverting. Confirmed via test: `uint128(2^128) == 0` with no revert. Added explicit `type(uint128).max` bounds checks for `cap`, `startPrice`, `endPrice`, `graduationTarget`, and `lpTokens` before storage writes. **Status: Fixed**
