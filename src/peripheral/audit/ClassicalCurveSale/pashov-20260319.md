## Findings

[80] **1. `buyExactIn` Caps Cost Below Actual Curve Price, Understating `raisedETH`**

`ClassicalCurveSale.buyExactIn` · Confidence: 80

**Description**
When the approximate token amount calculation in `buyExactIn` produces an `amount` whose exact curve cost slightly exceeds `netETH`, the code silently caps `cost = netETH` (line 600) and records this underpayment into `raisedETH`, permanently understating the contract's ETH obligations — meaning sellers may eventually receive an `InsufficientLiquidity` revert because `raisedETH` does not fully cover their expected proceeds.

**Fix**

```diff
         uint256 cost = _cost(startPrice, endPrice, vr, sold, amount);
-        if (cost > netETH) cost = netETH;
+        if (cost > netETH) {
+            // Approximation overshoot — reduce amount by 1 so cost fits within netETH
+            if (amount == 0) revert ZeroAmount();
+            unchecked { amount -= 1; }
+            cost = _cost(startPrice, endPrice, vr, sold, amount);
+        }
```

[80] **2. Fee-on-Transfer and Rebasing Token Accounting Mismatch**

`ClassicalCurveSale.configure` · Confidence: 80

**Description**
`configure()` records `c.cap` and `c.lpTokens` from the caller-supplied values, but `safeTransferFrom` only delivers what the token allows after its transfer fee — if the token charges a transfer fee or rebases downward, the contract's internal accounting (`cap`, `lpTokens`, `sold`) will exceed the actual token balance held, causing later `safeTransfer` calls in `buy()` or `graduate()` to revert once the shortfall is reached and permanently locking buyers out or corrupting LP seeding.

**Fix**

```diff
     _configure(...);

-    // Pull tokens from caller (cap for curve + lpTokens for LP reserve)
-    safeTransferFrom(token, address(this), cap + lpTokens);
+    // Measure actual received amount to guard against fee-on-transfer tokens
+    uint256 balBefore = ERC20(token).balanceOf(address(this));
+    safeTransferFrom(token, address(this), cap + lpTokens);
+    uint256 received = ERC20(token).balanceOf(address(this)) - balBefore;
+    if (received < cap + lpTokens) revert InvalidParams();
```

[75] **3. Creator-Controlled Fee Recipient Can Permanently DoS All Buys**

`ClassicalCurveSale.buy` / `buyExactIn` · Confidence: 75

**Description**
In `buy()` and `buyExactIn()`, the protocol performs `safeTransferETH(creator, fee)` unconditionally — if the creator uses `setCreator()` to set a contract that reverts on ETH receipt as the creator address, and `feeBps > 0`, every buy call on that token permanently reverts, locking all curve trading. The same pattern affects `graduate()` which refunds unused ETH to the creator.
