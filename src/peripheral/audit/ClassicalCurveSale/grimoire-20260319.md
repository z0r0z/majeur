## Findings

### [MEDIUM] `buyExactIn()` Approximation Overshoot Causes DoS

**File:** `ClassicalCurveSale.sol` lines 582-601
**Confidence:** High

**Description:**
The inverse curve formula `amount = netETH * R / (A + netETH)` (line 588) approximates how many tokens can be bought for a given ETH input. The exact cost `_cost(amount)` can exceed `netETH` due to the approximation overshoot combined with `mulDivUp` ceiling rounding in `_cost()`. The current fix subtracts 1 from `amount` and recomputes, but for curves with significant price slope, a single decrement is insufficient — the cost may still exceed `netETH`.

When `cost > netETH` persists after the fix, the refund calculation `msg.value - cost - fee` (line 617) underflows, causing the transaction to revert. This is a DoS (not fund loss) that affects `buyExactIn()` for valid, realistic curve configurations, particularly when `feeBps = 0`.

**Attack path:**
1. Creator launches token with `startPrice = 0.005e18`, `endPrice = 0.02e18`, `cap = 1000e18`, `feeBps = 0`
2. After some buys (`sold = 250e18`), user calls `buyExactIn` with `msg.value = 1 ETH`
3. Approximation computes `amount`, but `_cost(amount - 1)` still exceeds `netETH`
4. Transaction reverts — user must fall back to `buy()` (exact-out) for a worse UX

**Impact:** DoS of `buyExactIn()` for steep curves with zero trading fee. Users can work around via `buy()`, but the primary "spend exact ETH" UX path is broken.

**Recommended fix:** Replace single-decrement with a loop or use a tighter inverse formula:
```solidity
while (cost > netETH && amount > 0) {
    unchecked { amount -= 1; }
    cost = _cost(startPrice, endPrice, vr, sold, amount);
}
```

## Dismissed Findings (Familiar Triage)

<details>
<summary><strong>Dismissed: Creator Transfer Silently Transfers Vesting Rights</strong></summary>

**Sigil 3 raw finding:** `setCreator()` transfers vesting claim rights to the new creator. The new creator can call `claimVested()` for tokens allocated to the original creator.

**Familiar verdict: FALSE POSITIVE (by design)**

This is the standard semantics of an ownership transfer function. `setCreator()` explicitly transfers all creator privileges, analogous to `Ownable.transferOwnership()`. The original creator can claim vested tokens before transferring. There is no "silent" transfer — the function is called by the current creator making a conscious decision. Unvested tokens transferring with the role is the expected semantic.

</details>

<details>
<summary><strong>Dismissed: Post-Graduation Creator Fee Rug Pull</strong></summary>

**Sigil 3 raw finding:** Creator can set up to 10% fee on both buy and sell at any time post-graduation with no timelock.

**Familiar verdict: OVERSTATED → Informational (design tradeoff)**

Mitigating factors:
- Fees are hard-capped at 10% per direction (`MAX_CREATOR_FEE_BPS = 1000`)
- Fees are configurable from `launch()` time — not a new post-graduation capability
- Standard DeFi trust model: users trust the creator they buy from (similar to Curve/Balancer admin fees)
- Creator identity is public on-chain; users can inspect fee config before participating
- Users can exit via LP removal (not fee-gated)

A timelock would be a nice-to-have but its absence is a design tradeoff, not a vulnerability. The 10% bound prevents catastrophic rug pulls.

</details>

## Detailed Analysis

<details>
<summary><strong>Sigil 1 — Reentrancy + CEI Violations</strong></summary>

**Hypothesis:** Reentrancy vectors via `graduate()` (no lock modifier), creator ETH fee push, ERC777 hooks, `beforeAction()` callback, `multicall()` delegatecall, or cross-function reentrancy.

**Investigation:**
- **`graduate()` reentrancy:** `c.seeded = true` set BEFORE any external calls (line 730). Reentry to `graduate()` fails at `!c.graduated || c.seeded` check. Reentry to buy/sell fails on `c.graduated` check. Creator CAN trigger graduation from `buy()` fee callback, but state is consistent and ETH balances work out — economically neutral.
- **ETH fee push:** `buy()`/`sell()` send fee to creator after state updates with `lock` held. Creator callback cannot reenter locked functions. Unprotected functions (`graduate`, `claimVested`, `setCreator`) have independent state guards.
- **ERC777 hooks:** Tokens from `launch()` are plain ERC20 clones with no hooks. External tokens via `configure()` are guarded by the transient `lock`.
- **`beforeAction()`:** Stateless callback — only reads storage and returns fee value. No state mutations possible.
- **`multicall()`:** Non-payable prevents msg.value double-spend. Delegatecall preserves transient storage context. Sequential execution means lock acquired/released per sub-call.
- **Cross-function during `graduate()`:** `claimVested()` operates on independent `CreatorVest` storage. `setCreator()` is harmless because `graduate()` caches `creator` in a local variable (line 732).

**Verdict: DISMISSED** — zero residual reentrancy vectors.

</details>

<details>
<summary><strong>Sigil 2 — Economic Exploits + Curve Math Precision</strong></summary>

**Hypothesis:** Exploitable precision errors in XYK curve math, fee arithmetic, or economic flows.

**Investigation:**
- **`buyExactIn()` overshoot:** CONFIRMED — see finding above. Approximation overshoots by more than 1 token for steep curves.
- **`_cost()` overflow:** `startPrice * amount` with both uint128-origin values fits in uint256 (`(2^128-1)^2 < 2^256`). DISMISSED.
- **`sellExactOut()` precision:** Uses `mulDivUp` which rounds token amount UP (protocol-favorable). User protected by `maxTokens` slippage. DISMISSED.
- **Fee arithmetic (8 swap paths):** All paths verified: `tax + net == gross` holds, no refund underflow possible, no value extraction. DISMISSED.
- **Graduation ETH accounting:** `raisedETH` is internal accounting, not `address(this).balance`. Force-sent ETH is unrecoverable but doesn't corrupt state. DISMISSED.
- **Round-trip extraction:** Buy and sell use identical `_cost()` with `mulDivUp` (rounds up = protocol-favorable in both directions). After fees, user always suffers net loss. DISMISSED.
- **Virtual reserve edge cases:** `vr > cap` always holds. `rem = vr - sold > 0` at all valid states. `configure()` validates `vr <= uint128.max`. DISMISSED.

**Verdict: 1 CONFIRMED** — `buyExactIn` DoS via approximation overshoot.

</details>

## Post-Review Responses

### `buyExactIn()` overshoot — needs stronger fix

The single-decrement fix applied from the Pashov audit is insufficient. The approximation can overshoot by multiple tokens for steep curves. A proper fix should either:
1. Use a while loop to decrement until cost fits
2. Use binary search between 0 and the approximated amount
3. Compute a provably-tight lower bound for the inverse formula
