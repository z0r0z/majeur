# Tribute — Zellic V12 Audit (2026-03-15)

> Automated audit via Zellic V12 on commit `7a39232`.

---

## Finding #3 — Stale reference arrays cause enumeration DoS

- **Severity:** High
- **Validity:** Valid (Duplicate)

### Description

`proposeTribute` appends references to `daoTributeRefs` and `proposerTributeRefs` without pruning or deduplication. When a tribute is canceled or claimed, the primary `tributes` mapping is cleared but references remain. Repeated propose/cancel cycles accumulate unbounded stale entries, causing `getActiveDaoTributes` to grow linearly in gas cost and potentially return duplicate active entries.

The Zellic PoC confirms: after propose → cancel → re-propose for the same key, `getActiveDaoTributes` returns 2 entries (duplicates) instead of 1.

### Assessment

**Valid, accepted (view-only concern). Duplicate of winfunc #21/23.**

This affects only the `getActiveDaoTributes` view function used for off-chain discovery. The core escrow mechanics (`proposeTribute`, `cancelTribute`, `claimTribute`) are unaffected — they operate on the `tributes` mapping directly and are correctly guarded by reentrancy protection.

The Zellic-proposed patch (swap-and-pop cleanup in `cancelTribute`/`claimTribute`) is a reasonable gas-vs-storage trade-off but adds O(n) iteration to every cancel/claim for a benefit that only affects a view function. Off-chain indexers (events) are the recommended discovery path. **Not applied — accepted as view-only limitation.**

## Finding #1 — Fee-on-transfer tribute misaccounting

- **Severity:** Low
- **Validity:** Valid (novel)

### Description

`proposeTribute` records the user-supplied `tribAmt` as the escrowed amount after `safeTransferFrom`, without checking the actual balance delta. For fee-on-transfer (FOT) tokens, the contract receives less than `tribAmt` but records the full amount, creating an insolvency condition where later `cancelTribute`/`claimTribute` attempts to transfer more than available.

### Assessment

**Valid, accepted (by design). Novel finding for Tribute.**

This is a standard FOT token concern applicable to virtually all escrow contracts that don't use balance-before/after accounting. The design decision is intentional:

1. **Standard ERC20 assumption:** Tribute is designed for standard ERC20 tokens and ETH. FOT, rebasing, and other non-standard tokens are outside the intended scope.
2. **Gas efficiency:** Balance-delta accounting adds two `balanceOf` calls (~2600 gas each) to every `proposeTribute` — unnecessary overhead for the 99.9% case of standard tokens.
3. **Self-contained risk:** A proposer using a FOT token harms themselves (their cancel returns less) and potentially other proposers of the same FOT token. The DAO is not at risk since it can verify the token before claiming.
4. **No attacker profit:** The "attacker" deposits FOT tokens and can only withdraw what the contract actually holds — no external value extraction beyond the escrow pool for that specific token.

**Not applied — accepted as out-of-scope token class.**
