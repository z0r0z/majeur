## Review Summary

> **Two runs conducted. Run 1 (pre-instructions): 0 novel, 1 FP, 3 duplicates. Run 2 (post-instructions): 1 novel finding (LOW-1), 0 FP, full defense/invariant coverage.**
>
> The comparison validates the SECURITY.md instructions hypothesis: without explicit directives, GPT produced 0% signal. With the 2-round methodology, it verified all 7 defenses, all 6 invariants, correctly screened 6 known findings + 2 FP patterns, and found a genuine novel issue in the discovery layer.
>
> **Run 2 novel finding — [LOW-1] valid and novel.** Reusing a cancelled/claimed `(proposer, dao, tribTkn)` key causes old stale refs to resolve to the new active mapping slot, producing duplicate entries in paginated views. This is distinct from T-4 (unbounded growth / gas DoS) — it's a semantic correctness bug in the discovery API. The ref arrays have no per-offer nonce, so historical refs cannot distinguish which offer instance they belong to. Impact is view-only: no fund risk, but frontends or on-chain integrators counting offers will see inflated/duplicate results. **Added as T-7.**

#### [LOW-1] Reusing a cancelled/claimed key resurrects stale refs into duplicate "active" listings

**Severity:** Low
**Confidence:** 93
**Location:** `Tribute.sol:101-103, 116, 154, 208-217, 253-262`

**Description:**
Ref arrays carry `(proposer, tribTkn)` or `(dao, tribTkn)` but no per-offer nonce. When an offer is cancelled/claimed and the same `(proposer, dao, tribTkn)` key is reused for a new offer, old historical refs resolve to the new active mapping slot. `getActiveDaoTributes` and `getActiveProposerTributes` return the same live offer multiple times — once per historical ref that shares that key.

**Attack path:**
1. Proposer P calls `proposeTribute(D, T, 100, F, 10)` → ref pushed (index 0)
2. P calls `cancelTribute(D, T)` → mapping deleted, ref at index 0 retained
3. P calls `proposeTribute(D, T, 200, F, 20)` → ref pushed (index 1)
4. `getActiveDaoTributes(D, 0, 10)` → loop hits ref[0] and ref[1], both resolve to same active `tributes[P][D][T]`, both pass `tribAmt != 0` check → duplicate entries returned

**Disproof attempt:**
No nonce/version in `TributeOffer` or refs. No offer ID binding. No consumed marker on old refs. The only filter (`tribAmt != 0` at lines 209, 254) checks the current mapping slot, not the historical ref instance. Old refs are reactivated whenever the same key becomes active again.

> **Review: VALID AND NOVEL.** Confirmed by code trace. This is distinct from T-4 (unbounded growth / gas DoS). T-4 is about stale entries costing gas to skip. This is about the view functions returning **semantically incorrect** results — one real offer appearing N times where N is the number of historical propose/cancel cycles on that key. A proposer could spam cancel/repost to make their offer dominate paginated listings, crowding out others. Impact is view-only (no fund risk), but frontends or on-chain integrators counting available tributes for a DAO will see inflated results. **Added as T-7.** Confidence 93 is appropriate.
