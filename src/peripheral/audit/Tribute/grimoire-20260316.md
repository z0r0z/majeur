## Finding Responses

### [I-01] CEI Violation in `proposeTribute` — Interaction Before Effects

**Severity:** Informational | **Confidence:** High | **Familiar verdict:** Confirmed (no exploit path)
**Location:** `Tribute.sol:91–105`

`safeTransferFrom(tribTkn, address(this), tribAmt)` at line 91 executes before state writes at lines 96–105. An ERC777 `tokensToSend` hook on the proposer fires mid-execution when `offer.tribAmt` is still 0, theoretically allowing the duplicate-offer guard to be bypassed.

**Blocked by:** `nonReentrant` modifier (transient storage guard). Any reentrant call during the hook reverts with `Reentrancy()`. The pattern is latent — becomes exploitable only if the guard is removed in a future refactor.

**Recommendation:** Move `safeTransferFrom` after state writes for defense-in-depth. Zero cost, eliminates the latent pattern entirely.

**Status:** Informational — no exploitable path exists.

### [I-02] ETH Push to Proposer in `claimTribute` — Reverting Receive DoS

**Severity:** Informational (context-adjusted) | **Confidence:** High | **Familiar verdict:** Confirmed, severity adjusted
**Location:** `Tribute.sol:157`

When `forTkn == address(0)`, `claimTribute` pushes ETH to the proposer via `safeTransferETH(proposer, offer.forAmt)`. If the proposer is a contract whose `receive()` reverts, the DAO cannot complete the swap. The proposer retains cancel ability.

**Why informational in context:** The intended DAO callers are Moloch contracts which have `receive() external payable {}`. The griefing scenario (proposer deploys reverting contract, DAO wastes governance time, proposer cancels) results in no fund loss — the revert is atomic and the DAO's ETH is never spent. The proposer must sacrifice gas to execute the grief.

**Status:** Acknowledged — documented in natspec. N/A to Moloch integration.

---

### [I-03] Unbounded Ref Array Growth

**Severity:** Informational (mitigated) | **Confidence:** High | **Familiar verdict:** Confirmed, severity adjusted from Medium to Informational
**Location:** `Tribute.sol:101–103`

`daoTributeRefs` and `proposerTributeRefs` are append-only. Repeated propose+cancel cycles grow the arrays permanently. View functions must scan stale entries.

**Mitigated by:** Pagination (`start`/`limit`/`next` pattern with OOB guard). Spam requires gas and token approvals. No on-chain state-changing function iterates these arrays.

**Status:** Mitigated — pagination added.

---

### [I-04] `limit=0` Pagination Ambiguity

**Severity:** Informational | **Confidence:** Medium | **Familiar verdict:** Confirmed, description imprecise
**Location:** `Tribute.sol:202–227`

When `limit=0` and `start=0`, the loop never runs and `next` returns 0 — indistinguishable from "no more results." For `start > 0`, `next` correctly returns `start` (not 0), so the ambiguity is limited to the `start=0` case.

**Status:** Informational — callers should pass `limit >= 1`.
