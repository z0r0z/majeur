## Finding #7 — Caller-supplied deadline bypasses sale end gate

- **Severity:** Critical
- **Validity:** Invalid

### Description

Zellic reports that `burnUnsold` authorizes burning based solely on a caller-supplied `deadline`, allowing any caller to pass an already-expired timestamp to burn shares prematurely.

### Assessment

**Invalid.** The audit misunderstands the permit-based architecture.

`burnUnsold` is never called directly by an external user. It runs via `delegatecall` through the Moloch permit system:

1. **Setup:** SafeSummoner encodes the `deadline` and `shares` address into a one-shot permit via `setPermit(1, burner, 0, burnData, salt, burner, 1)` during DAO initialization.
2. **Execution:** `closeSale(dao, shares, deadline, nonce)` calls `dao.spendPermit(...)`, which verifies the encoded calldata (including `deadline`) matches the stored permit exactly.
3. **One-shot:** The permit count is 1 — it can only be spent once.

The caller-supplied `deadline` in `closeSale` must match the permit's encoded deadline set at deployment. A mismatched deadline causes `spendPermit` to revert. The `burnUnsold` function itself does not need access control because it only executes within the DAO's delegatecall context after permit verification.

**No fix needed.**
