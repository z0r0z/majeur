### Round 1 — Defense Verification

**msg.sender keying** — `configure()` writes `sales[msg.sender]` (line ~58). No other function writes to `sales`. **Verified.**

**Checked arithmetic** — `cost = amount * s.price / 1e18` appears at lines ~72 and ~109, both outside `unchecked` blocks. The only `unchecked` blocks wrap `msg.value - cost`, which is guarded by `if (msg.value > cost)` — safe. **Verified.**

**ETH guard** — Line ~83: `if (msg.value != 0) revert UnexpectedETH()` in the `else` (ERC20) branch of `buy()`. `buyExactIn()` reverts early if `s.payToken != address(0)` (line ~97), so it never reaches an ERC20 path. **Verified.**

**CEI ordering** — `spendAllowance` is called at line ~75 in `buy()` and line ~107 in `buyExactIn()`, both *before* any `safeTransferETH`, `safeTransferFrom`, or `safeTransfer` calls. All external interactions follow. **Verified.**

**Zero-cost guard** — `if (cost == 0) revert ZeroAmount()` at line ~73 (`buy()`) and line ~105 (`buyExactIn()`). **Verified.**

**Deadline enforcement** — `if (s.deadline != 0 && block.timestamp > s.deadline) revert Expired()` at line ~66 (`buy()`) and line ~98 (`buyExactIn()`). Deadline of 0 disables the check. Inclusive — purchases at exactly `deadline` are allowed. **Verified.**

**Allowance cap** — Both functions read `remaining` from `IMoloch(dao).allowance()`, cap `amount` with `if (amount > remaining) amount = remaining`, and revert if `amount == 0`. **Verified** (with documented KF#6 caveat for fake DAOs).

**ZeroPrice guard** — `configure()` reverts on `price == 0` (line ~57). Both buy functions revert on `s.price == 0` as a "not configured" sentinel. **Verified.**

**Moloch nonReentrant** — Cross-contract dependency, not enforceable within this file. Documented as assumption. **Verified as documented.**

### Round 2 — Adversarial Hunt

I examined the following attack vectors and found **no novel findings** surviving all checks:

**Candidate: Front-running `configure()` price changes.** A pending governance call to raise the price could be front-run by a buyer at the old price. Checked against DC-6 — this is a documented design choice (no price-lock mechanism). **Discarded.**

**Candidate: TOCTOU between `allowance()` read and `spendAllowance()`.** Within a single transaction, these are atomic. Across transactions, if a concurrent buyer reduces the allowance between the view call and `spendAllowance`, the Moloch call simply reverts — no fund loss, just a failed tx. Standard first-come-first-served behavior. **Discarded — no impact.**

**Candidate: `safeTransferETH` forwards all gas to refund recipient.** After the CEI fix, `spendAllowance` has already decremented the allowance before the refund callback. Re-entering `buy()` reads the updated (lower) allowance. ShareSale has no per-user mutable state to corrupt. **Discarded — matches dismissed Grimoire Sigil 1 finding.**

**Candidate: Deadline boundary — `block.timestamp == deadline` allows purchase.** The check is `>`, not `>=`, making the deadline inclusive. This is conventional and consistent with how most protocols treat deadlines. **Discarded — standard behavior, not a vulnerability.**

**Candidate: Assembly memory safety in free functions.** `safeTransferFrom` saves and restores the free memory pointer (`0x40`) and clears `0x60`. `safeTransfer` clears `0x34`. Both follow standard Solady patterns and are marked `memory-safe`. **Discarded — no corruption found.**

The post-patch code is clean. All 9 defense mechanisms verified. All 6 invariants hold. No novel findings survived the adversarial hunt. The 6 known findings are appropriately addressed (5 patched, 1 accepted with documented rationale).
