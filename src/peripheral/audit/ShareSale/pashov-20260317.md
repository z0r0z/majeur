## Findings

[100] **1. Reentrancy in `buy()` via ETH Refund Before `spendAllowance`**

`ShareSale.buy` · Confidence: 100

**Description**
`buy()` sends the ETH refund to `msg.sender` (line 84) before calling `IMoloch(dao).spendAllowance()` (line 93), so a buyer contract whose `receive()` re-enters `buy()` with the refunded ETH can purchase shares a second time against the still-unreduced allowance, double-spending the DAO's share allowance. The same pattern exists in `buyExactIn()` (refund at line 130, spendAllowance at line 134).

**Fix**

```diff
+       // Spend allowance first (CEI: effects before interactions)
+       IMoloch(dao).spendAllowance(s.token, amount);
+
        // Collect payment
        if (s.payToken == address(0)) {
            if (msg.value < cost) revert InsufficientPayment();
            safeTransferETH(dao, cost);
            if (msg.value > cost) {
                unchecked {
                    safeTransferETH(msg.sender, msg.value - cost);
                }
            }
        } else {
            if (msg.value != 0) revert UnexpectedETH();
            safeTransferFrom(s.payToken, dao, cost);
        }

-       // Spend allowance — _payout mints/transfers to this contract
-       IMoloch(dao).spendAllowance(s.token, amount);
```

**Response:** Valid. CEI pattern violated — external calls (ETH transfers) happen before the state-changing `spendAllowance` call. Both `buy()` and `buyExactIn()` affected. **Status: Patched** — moved `spendAllowance` before all payment interactions.

[85] **2. Zero-Cost Share Acquisition via Price Rounding to Zero**

`ShareSale.buy` · Confidence: 85

**Description**
When `price < 1e18` (e.g., `price = 1e16` for 0.01 ETH/share), buying small amounts causes `cost = amount * price / 1e18` to truncate to zero; the `msg.value < cost` guard passes when cost is 0, allowing a buyer to acquire small amounts of shares for free. Practically limited by gas costs (each call yields only dust amounts of shares), but a `cost == 0` check is cheap insurance.

**Fix**

```diff
        uint256 cost = amount * s.price / 1e18;
+       if (cost == 0) revert ZeroAmount();

        // Collect payment
```

**Response:** Valid but low practical impact. Exploitation requires ~1e16 calls per whole share at typical prices, making it gas-prohibitive. Still worth the guard as defense-in-depth. **Status: Patched** — added `cost == 0` revert in both `buy()` and `buyExactIn()`.

[85] **3. ERC777 `tokensToSend` Reentrancy in `buy()` via ERC20 Payment Path**

`ShareSale.buy` · Confidence: 85

**Description**
In the ERC20 payment path, `safeTransferFrom(s.payToken, dao, cost)` is called before `IMoloch(dao).spendAllowance()`, so if `payToken` is an ERC777-compatible token, the buyer's `tokensToSend` hook fires mid-transfer and can re-enter `buy()` to purchase additional shares against the still-unreduced allowance. Same root cause as Finding 1 — CEI violation — and resolved by the same fix (moving `spendAllowance` before payment collection).

**Fix**

Same as Finding 1 — move `spendAllowance` before all external interactions.

**Response:** Valid. Same root cause as Finding 1. Requires ERC777-compatible payToken which limits scope. **Status: Patched** — resolved by same CEI fix as Finding 1.
