## Methodology

**Grimoire** is a workflow meta-toolkit for security research. Unlike scanner-style tools, it uses specialized agent roles:

- **Sigil agents** — single-context hunters that receive one attack surface hypothesis and search the codebase for evidence. Each sigil tries to PROVE a vulnerability exists.
- **Familiar agents** — adversarial verifiers that independently read the code and try to DISPROVE each sigil finding. Whatever survives both passes is worth the researcher's time.

## Detailed Analysis

<details>
<summary><strong>Sigil 1 — Reentrancy + CEI Violations</strong></summary>

**Hypothesis:** Even after the CEI fix (KF#3/KF#5), reentrancy vectors may remain via `spendAllowance` callbacks, cross-function reentrancy after ETH refund/ERC777 hooks, or share token transfer hooks.

**Investigation:**
- **spendAllowance re-entry:** Moloch's `spendAllowance` is `nonReentrant`. Its internal `_payout` mints shares/loot to ShareSale, triggering `onSharesChanged` callbacks within trusted DAO contracts only (not attacker-controllable).
- **Post-spendAllowance callbacks:** ETH refund (line 88) and ERC777 `tokensToSend` (line 93) are attacker-controllable. However, at these points `spendAllowance` has already decremented the allowance. Re-entering `buy()` reads the updated (lower) allowance — no double-spend possible.
- **Share transfer hooks:** Shares/loot tokens are custom Moloch tokens without ERC777-style hooks to arbitrary addresses. Transfer at line 105 is the final interaction.
- **Stateless singleton:** ShareSale holds no mutable per-user state. Only `sales` mapping exists, keyed by DAO address and writable only by the DAO itself.

**Verdict: DISMISSED** — zero residual reentrancy vectors.

</details>

<details>
<summary><strong>Sigil 2 — Economic Exploits + Precision/Rounding</strong></summary>

**Hypothesis:** Round-trip precision loss in `buyExactIn()`, fee-on-transfer token issues, overflow DoS, large-refund arithmetic errors, or price manipulation.

**Investigation:**
- **Round-trip precision:** `amount = msg.value * 1e18 / s.price` → `cost = amount * s.price / 1e18`. Max underpayment per call is `(s.price - 1) / 1e18`, which is < 1 wei for typical prices. Not exploitable.
- **Fee-on-transfer:** `safeTransferFrom(s.payToken, dao, cost)` — DAO receives less than `cost`. However, the DAO itself chooses `payToken` via `configure()`. Configuration concern, not contract vulnerability.
- **Overflow DoS:** `amount * s.price` with Solidity 0.8 checked math. Overflow requires values far beyond total ETH supply (~1.2e26 wei). Unreachable in practice.
- **Refund arithmetic:** In `buyExactIn`, after capping `amount = remaining`, mathematically `remaining * s.price / 1e18 < msg.value` (proven from the cap condition). Unchecked subtraction is safe.
- **Price front-running:** `configure()` keyed to `msg.sender` — only the DAO (via governance) can change price. Not front-runnable.

**Verdict: DISMISSED** — all economic vectors bounded by dust or by-design.

</details>

<details>
<summary><strong>Sigil 3 — Access Control + Privilege Boundaries</strong></summary>

**Hypothesis:** Fake sale config social engineering, third-party sale disruption, `saleInitCalls()` injection, buying from unconfigured DAOs, stuck funds, deadline bypass, cross-DAO interference.

**Investigation:**
- **Fake config:** An attacker can only write `sales[attacker_address]`. Buyer must explicitly pass attacker's address as `dao`. Attacker's contract must implement full IMoloch interface. Not a ShareSale flaw.
- **Third-party disruption:** Only ShareSale can spend its own allowance (`allowance[token][ShareSale]`). Third parties cannot drain it. Permissionless buying at configured price is by design.
- **saleInitCalls injection:** Pure `view` function generating ABI-encoded calldata. No side effects. Type-safe encoding prevents injection.
- **Unconfigured DAO:** `price == 0` check at line 68 correctly catches uninitialized structs.
- **Stuck funds:** No `receive()`/`fallback()`. ETH flows through transiently. Patched ETH guard (KF#2) prevents stray ETH.
- **Deadline bypass:** `block.timestamp > deadline` check is correct. `deadline == 0` means no expiry (documented).
- **Cross-DAO:** `sales` mapping keyed by DAO address. Fully isolated per-DAO state.

**Verdict: DISMISSED** — access model is sound.

</details>

<details>
<summary><strong>Sigil 4 — Assembly Correctness + Token Compatibility</strong></summary>

**Hypothesis:** `codesize()` misuse, memory corruption, `caller()` in delegatecall contexts, return value handling edge cases, memory safety asymmetry, token compatibility, error selector correctness.

**Investigation:**
- **`codesize()` in `safeTransferETH`:** Used as offset with length 0. Zero length means offset is irrelevant. Standard gas optimization (2 gas vs 3 for PUSH).
- **Memory cleanup:** `safeTransfer` cleans `mstore(0x34, 0)` — restores FMP region. `safeTransferFrom` saves/restores FMP and zeros 0x60. Asymmetry is correct — `safeTransferFrom` touches more memory.
- **`caller()` usage:** ShareSale is a singleton, never delegatecalled. `caller()` correctly references `msg.sender`.
- **Return value handling:** Verified all 5 scenarios (returns true, no return, returns false, reverts, no code). Byte-identical to Solady SafeTransferLib.
- **Error selectors:** `0xb12d13eb` = `ETHTransferFailed()`, `0x90b8ec18` = `TransferFailed()`, `0x7939f424` = `TransferFromFailed()`. All confirmed via `cast sig`.
- **Token compatibility:** USDT (no return) handled by `returndatasize()=0` path. USDC (returns true) handled by `eq(mload(0x00), 1)`. Direct comparison with Solady source confirmed byte-identical patterns.

**Verdict: DISMISSED** — assembly is correct and battle-tested.

</details>
