## Round 1 — Defense Verification

### Defenses

**1) msg.sender keying — Verified**
**Evidence:** `configure()` writes only `sales[msg.sender]` and emits `Configured(msg.sender, ...)`; it never writes a third party’s slot.
**Location:** `src/peripheral/ShareSale.sol:53-56`
**Conclusion:** **Verified.** This enforces self-keying per caller address. It does **not** prove the caller is a genuine Moloch DAO; it only ensures an address can configure only its own entry.

**2) Checked arithmetic — Verified**
**Evidence:** Both pricing computations use checked Solidity 0.8 arithmetic and are not inside an `unchecked` block: `amount * s.price / 1e18` in `buy()` and `msg.value * 1e18 / s.price`, then `amount * s.price / 1e18` in `buyExactIn()`.
**Location:** `src/peripheral/ShareSale.sol:76-77,120,127-128`
**Conclusion:** **Verified.** KF#1’s unchecked-wraparound path is gone.

**3) ETH guard — Verified**
**Evidence:** In the ERC20 payment branch of `buy()`, nonzero `msg.value` reverts with `UnexpectedETH()`. `buyExactIn()` also rejects non-ETH sales up front.
**Location:** `src/peripheral/ShareSale.sol:91-93,115-116`
**Conclusion:** **Verified.** KF#2’s stray-ETH-on-ERC20-purchase path is blocked.

**4) CEI ordering — Verified**
**Evidence:** `spendAllowance()` runs before all payment interactions and before the final token transfer in both buy paths.
**Location:** `src/peripheral/ShareSale.sol:79-80,82-94,96-105,130-148`
**Conclusion:** **Verified.** The patched order matches CEI and removes the pre-payment callback window that enabled KF#3/KF#5.

**5) Zero-cost guard — Verified**
**Evidence:** Both buy paths revert if `cost == 0`.
**Location:** `src/peripheral/ShareSale.sol:76-77,127-128`
**Conclusion:** **Verified.** KF#4’s zero-cost purchase path is blocked.

**6) Deadline enforcement — Verified**
**Evidence:** Both `buy()` and `buyExactIn()` revert when `block.timestamp > s.deadline` and `deadline != 0`.
**Location:** `src/peripheral/ShareSale.sol:68-69,115-117`
**Conclusion:** **Verified.** Purchases after expiry are rejected. Purchases at the exact deadline timestamp are still allowed because the check is `>` rather than `>=`.

**7) Allowance cap — Violated**
**Evidence:** `ShareSale` caps `amount` to whatever `IMoloch(dao).allowance(...)` reports, then calls `IMoloch(dao).spendAllowance(...)`, but it does not authenticate `dao` as a real Moloch or verify that `spendAllowance()` actually reduced a real allowance or delivered fresh tokens.
**Location:** `src/peripheral/ShareSale.sol:72-80,123-131,176-181`
**Conclusion:** **Violated.** The local min-cap logic exists, but the advertised defense “cannot exceed DAO-approved cap” is not enforced by `ShareSale` itself against arbitrary `dao` contracts.

**8) ZeroPrice guard — Verified**
**Evidence:** `configure()` reverts on `price == 0`. Unconfigured sales also fail because `buy()`/`buyExactIn()` require `s.price != 0`.
**Location:** `src/peripheral/ShareSale.sol:53-56,67-69,114-117`
**Conclusion:** **Verified.** This prevents zero-price configuration and blocks zero-initialized structs from masquerading as valid sales.

**9) Moloch nonReentrant — Violated (in-scope)**
**Evidence:** The supplied file only declares the `IMoloch` interface; no `nonReentrant` implementation is present or enforceable from this file.
**Location:** `src/peripheral/ShareSale.sol:176-181`

## Round 2 — Adversarial Hunt

I looked for issues not covered by the Known Findings or the Design Choices tables, then tried to disprove each one. One finding survived.

### [LOW-1] Fake DAO can drain unsolicited ERC20 balances held by the singleton

* **Severity:** Low
* **Confidence:** 88
* **Location:** `src/peripheral/ShareSale.sol:53-56,72-80,97-105,123-131,140-148,176-181`

**Description**
`ShareSale` assumes that `dao` is a genuine Moloch implementing honest allowance semantics. That assumption is never enforced. The contract will:

1. trust `dao.allowance(...)` as the cap source,
2. trust `dao.spendAllowance(...)` to actually fund `ShareSale`, and then
3. transfer tokens to the buyer from `ShareSale`’s own balance.

Because ERC20 tokens can be sent directly to any contract, the singleton can accumulate unsolicited token balances. An attacker can then deploy a fake `dao` contract whose `allowance()` returns a large number and whose `spendAllowance()` is a no-op, configure a sale for that fake DAO, and use `buy()` or `buyExactIn()` to make `ShareSale` transfer out those preexisting tokens.

The attacker does not need to suffer a real payment loss if the sale is ETH-priced, because the “payment” is sent to the attacker-controlled fake DAO and can be withdrawn back.

This is not KF#1-5, and it is not covered by DC-1..DC-7.

**Attack path**

1. Some ERC20 tokens are accidentally transferred to `ShareSale`.
2. Attacker deploys a fake DAO implementing:

   * `allowance(...)` → returns a large value,
   * `spendAllowance(...)` → no-op,
   * `shares()/loot()` → irrelevant if `token` is configured directly as the target ERC20.
3. The fake DAO calls `ShareSale.configure(targetToken, address(0), 1e18, 0)`, which writes `sales[fakeDao]`.
4. The attacker calls `buyExactIn(fakeDao)` with `msg.value = N`.
5. `ShareSale` computes `amount = N`, calls fake `spendAllowance()` (no-op), sends `N` wei to the fake DAO, then transfers `N` units of `targetToken` from its own existing balance to the attacker.
6. The attacker withdraws the ETH from the fake DAO. Net cost is essentially gas.

**Disproof attempt**

* If `ShareSale` could never hold ERC20 balances, there would be nothing to steal. That assumption is false: ERC20s can be sent directly to any contract, and there is no sweep/recovery path here.
* If `dao` were guaranteed to be a trusted Moloch implementation, the fake-DAO leg would fail. That assumption is also unenforced in `ShareSale`.
* This does not let an attacker steal ETH already in `ShareSale` by the same mechanism; the payout leg is an ERC20 transfer. The issue is specifically about unsolicited ERC20 balances.

**Recommendation**
Authenticate the `dao` dependency or verify fresh funding before forwarding tokens:

* safest: restrict `dao` to trusted Moloch instances from a known factory/registry;
* additionally or alternatively: resolve the outgoing token address, record `balanceBefore`, call `spendAllowance`, then require `balanceAfter >= balanceBefore + amount` before payment/forwarding;
* add a governance-controlled rescue function for unsolicited ERC20/forced-ETH balances.
